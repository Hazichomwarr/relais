# KARMDA

# Formation

## Documentation RELAIS

> Ce dossier contient la méthode officielle de formation des utilisateurs de KARMDA.

---

# Bienvenue

La réussite de KARMDA dépend autant de la capacité des utilisateurs à l’utiliser que de la qualité de la plateforme elle-même.

Une fonctionnalité peut être parfaitement conçue, correctement configurée et techniquement disponible.

Si les utilisateurs ne savent pas :

- pourquoi elle existe ;
- quand l’utiliser ;
- comment l’utiliser ;
- comment vérifier leur travail ;
- quoi faire en cas de difficulté ;

elle ne produira pas les résultats attendus.

La formation transforme les fonctionnalités de KARMDA en pratiques réelles dans l’établissement.

---

# Notre philosophie

Chez RELAIS, former ne consiste pas à montrer toutes les pages de KARMDA.

Former consiste à permettre à une personne de réaliser correctement les tâches liées à son rôle.

Un enseignant n’a pas besoin de connaître toute la plateforme.

Il doit surtout savoir :

- se connecter ;
- retrouver ses classes ;
- enregistrer les présences ;
- créer ou consulter une évaluation ;
- saisir les notes ;
- vérifier son travail ;
- demander de l’aide.

Un administrateur n’a pas besoin d’apprendre KARMDA comme un développeur.

Il doit savoir gérer les opérations administratives qui lui sont confiées.

> **La formation ne doit pas suivre la structure du logiciel. Elle doit suivre le travail de l’utilisateur.**

---

# Pourquoi ce dossier est important

Une formation insuffisante peut provoquer :

- des erreurs de saisie ;
- des données incomplètes ;
- une mauvaise utilisation des rôles ;
- une dépendance excessive envers RELAIS ;
- un retour aux anciens processus ;
- une perte de confiance ;
- des utilisateurs qui affirment que le système ne fonctionne pas alors qu’ils ne maîtrisent pas encore la procédure ;
- une adoption faible malgré un déploiement techniquement réussi.

À l’inverse, une bonne formation permet :

- une prise en main plus rapide ;
- une meilleure qualité des données ;
- une réduction des demandes de support ;
- une plus grande autonomie ;
- une meilleure adoption ;
- une utilisation cohérente dans tout l’établissement.

---

# À qui s’adresse ce dossier ?

Cette documentation est destinée :

- aux formateurs RELAIS ;
- aux chefs de projet ;
- aux responsables du déploiement ;
- aux responsables internes des établissements ;
- aux ambassadeurs KARMDA ;
- aux personnes chargées d’organiser des rappels de formation.

---

# Les objectifs de la formation

Une formation KARMDA doit permettre à chaque utilisateur de :

1. comprendre le rôle de KARMDA dans son travail ;
2. accéder correctement à son compte ;
3. réaliser les tâches essentielles liées à son rôle ;
4. reconnaître les erreurs fréquentes ;
5. vérifier les informations saisies ;
6. respecter les règles de sécurité ;
7. savoir quand demander de l’aide ;
8. reproduire les actions sans assistance constante.

Le véritable objectif n’est pas que l’utilisateur dise :

> J’ai assisté à la formation.

L’objectif est qu’il puisse dire :

> Je sais réaliser mon travail dans KARMDA.

---

# Les principes de formation RELAIS

## 1. Former par rôle

Les utilisateurs ne doivent pas tous recevoir exactement la même formation.

Chaque rôle possède :

- ses responsabilités ;
- ses permissions ;
- ses tâches ;
- ses risques ;
- ses besoins.

Les formations doivent donc être adaptées aux profils suivants :

- administrateurs ;
- direction ;
- personnel administratif ;
- enseignants ;
- parents ;
- élèves ;
- ambassadeurs internes.

---

## 2. Former à partir des tâches réelles

Les exemples doivent correspondre aux activités quotidiennes de l’établissement.

Par exemple :

- inscrire un nouvel élève ;
- affecter un élève à une classe ;
- enregistrer une présence ;
- créer une évaluation ;
- saisir une note ;
- enregistrer un paiement ;
- consulter un bulletin ;
- publier une annonce ;
- retrouver un rapport.

Une démonstration déconnectée du travail réel est rapidement oubliée.

---

## 3. Expliquer avant de démontrer

Avant de montrer une action, le formateur doit expliquer :

- pourquoi elle est réalisée ;
- qui doit la réaliser ;
- à quel moment ;
- quelles informations sont nécessaires ;
- quelles conséquences elle produit.

L’utilisateur comprend ainsi le processus, pas seulement les clics.

---

## 4. Montrer, faire pratiquer, vérifier

Chaque compétence importante doit suivre cette séquence :

```text
Expliquer
    ↓
Démontrer
    ↓
Faire pratiquer
    ↓
Observer
    ↓
Corriger
    ↓
Faire recommencer
    ↓
Valider
```

Voir une action ne signifie pas savoir la reproduire.

---

## 5. Utiliser les données de l’établissement

Lorsque cela est possible, les exercices doivent utiliser :

- les vraies classes ;
- les vrais rôles ;
- des scénarios proches de la réalité ;
- des données de test clairement identifiées ;
- les processus propres à l’établissement.

Les utilisateurs apprennent plus facilement lorsqu’ils reconnaissent leur environnement.

Les données sensibles ne doivent toutefois jamais être exposées inutilement.

---

## 6. Former progressivement

Une formation trop dense réduit la mémorisation.

Il vaut mieux enseigner :

- les tâches essentielles avant les fonctions avancées ;
- les processus quotidiens avant les cas exceptionnels ;
- quelques actions maîtrisées plutôt qu’un grand nombre de fonctions survolées.

La progression recommandée est :

```text
Comprendre
    ↓
Se connecter
    ↓
Naviguer
    ↓
Réaliser une tâche simple
    ↓
Réaliser un processus complet
    ↓
Corriger une erreur
    ↓
Travailler en autonomie
```

---

## 7. Laisser une trace

Chaque formation doit laisser aux utilisateurs des ressources qu’ils peuvent consulter après la session.

Par exemple :

- guide utilisateur ;
- fiche pratique ;
- checklist ;
- vidéo courte ;
- captures d’écran ;
- procédure de support ;
- questions fréquentes ;
- exercices.

Une formation uniquement orale disparaît rapidement de la mémoire.

---

## 8. Vérifier la compétence

La participation ne constitue pas une validation.

Pour confirmer qu’un utilisateur maîtrise une tâche, il doit la réaliser lui-même.

Par exemple, un enseignant formé doit pouvoir :

- se connecter ;
- retrouver sa classe ;
- ouvrir une session de présence ;
- enregistrer les statuts ;
- vérifier les résultats ;
- corriger une erreur selon ses permissions.

La compétence doit être observée.

---

## 9. Ne pas humilier l’apprenant

Les utilisateurs possèdent des niveaux de familiarité numérique différents.

Certains apprendront rapidement.

D’autres auront besoin :

- de plus de temps ;
- de répétitions ;
- d’exemples ;
- d’une explication différente ;
- d’un accompagnement individuel.

Le formateur ne doit jamais :

- ridiculiser une question ;
- montrer de l’impatience ;
- comparer publiquement les participants ;
- prendre le contrôle trop rapidement ;
- faire croire qu’une difficulté est évidente.

> **Une formation réussie crée de la confiance avant de créer de la vitesse.**

---

# Le cycle de formation KARMDA

La formation doit être considérée comme un processus.

```text
Identifier les utilisateurs
        ↓
Analyser leurs tâches
        ↓
Préparer les supports
        ↓
Organiser la session
        ↓
Expliquer et démontrer
        ↓
Faire pratiquer
        ↓
Évaluer
        ↓
Corriger les difficultés
        ↓
Assurer un suivi
```

La session en salle ou en ligne n’est qu’une partie du processus.

---

# Avant la formation

Avant chaque session, le formateur doit connaître :

- les participants ;
- leurs rôles ;
- leur niveau d’expérience ;
- les tâches qu’ils devront réaliser ;
- les modules activés ;
- les appareils qu’ils utiliseront ;
- la langue de formation ;
- la qualité de la connexion ;
- les besoins particuliers ;
- les problèmes déjà identifiés.

Il doit aussi vérifier :

- que les comptes sont créés ;
- que les permissions sont correctes ;
- que les mots de passe temporaires fonctionnent ;
- que les données nécessaires sont disponibles ;
- que l’environnement de formation est prêt ;
- que le matériel fonctionne ;
- que les supports sont accessibles.

Une session ne doit pas perdre sa première heure à créer les comptes qui auraient dû être préparés.

---

# Pendant la formation

Le formateur doit :

- présenter les objectifs ;
- expliquer le déroulement ;
- rappeler que les questions sont encouragées ;
- utiliser un langage simple ;
- relier chaque action au travail réel ;
- avancer étape par étape ;
- observer les participants ;
- laisser du temps à la pratique ;
- corriger immédiatement les erreurs importantes ;
- noter les questions récurrentes ;
- vérifier régulièrement la compréhension.

## Questions utiles

- Que souhaitez-vous accomplir avec cette action ?
- Où retrouveriez-vous cette information ?
- Quelle étape vient ensuite ?
- Comment vérifieriez-vous que l’opération a réussi ?
- Que feriez-vous si cette information était incorrecte ?
- Qui contacteriez-vous si vous étiez bloqué ?

---

# Après la formation

Après la session, RELAIS doit :

- transmettre les supports ;
- enregistrer les présences ;
- documenter les difficultés ;
- identifier les personnes nécessitant un accompagnement supplémentaire ;
- transmettre les questions non résolues ;
- organiser une session de rattrapage lorsque nécessaire ;
- évaluer les compétences ;
- suivre l’utilisation réelle ;
- mettre à jour les supports selon les retours.

La formation ne doit pas être considérée comme terminée le jour où le formateur quitte la salle.

---

# Formats de formation

La formation peut prendre plusieurs formes.

## Formation en présentiel

Adaptée lorsque :

- plusieurs utilisateurs se trouvent dans le même établissement ;
- la connexion est limitée ;
- les participants ont besoin d’un accompagnement rapproché ;
- des exercices pratiques doivent être observés directement.

## Formation à distance

Adaptée lorsque :

- les participants sont dispersés ;
- le nombre d’utilisateurs est limité ;
- la connexion est suffisamment stable ;
- le partage d’écran est possible.

## Formation en petits groupes

Recommandée lorsque les rôles sont différents.

Par exemple :

- administration ;
- enseignants ;
- direction ;
- finance.

## Formation individuelle

Utile pour :

- un responsable clé ;
- un nouvel utilisateur ;
- une personne ayant besoin de rattrapage ;
- une fonction sensible ;
- une difficulté particulière.

## Autoformation accompagnée

Peut inclure :

- documentation ;
- vidéos ;
- exercices ;
- quiz ;
- session de questions.

Elle ne doit pas être utilisée seule lorsque les tâches sont critiques ou complexes.

---

# Durée des sessions

La durée doit être adaptée au contenu et au niveau des participants.

À titre indicatif :

| Public                |                        Durée indicative |
| --------------------- | --------------------------------------: |
| Direction             |                            1 à 2 heures |
| Administration        |                            2 à 4 heures |
| Enseignants           |                       1 h 30 à 3 heures |
| Finance               |                            2 à 3 heures |
| Parents               |                         30 à 60 minutes |
| Élèves                |                         30 à 60 minutes |
| Ambassadeurs internes | Session utilisateur + approfondissement |

Une formation longue n’est pas nécessairement une meilleure formation.

Lorsque le contenu est important, il peut être préférable d’organiser plusieurs sessions courtes.

---

# Taille des groupes

La taille du groupe doit permettre la pratique et l’observation.

## Petit groupe

De 5 à 15 participants.

Avantages :

- échanges faciles ;
- observation précise ;
- correction rapide ;
- rythme adaptable.

## Groupe moyen

De 15 à 30 participants.

Nécessite :

- une organisation stricte ;
- des assistants éventuels ;
- des exercices bien préparés ;
- un temps suffisant.

## Grand groupe

Au-delà de 30 participants.

Une grande présentation peut servir à introduire le projet.

Elle ne remplace généralement pas les ateliers pratiques en petits groupes.

---

# Préparer un plan de session

Chaque formation doit posséder un plan.

## Exemple

| Étape             | Contenu                          |  Durée |
| ----------------- | -------------------------------- | -----: |
| Introduction      | Objectifs et rôle de KARMDA      | 10 min |
| Connexion         | Accès et sécurité                | 15 min |
| Navigation        | Tableau de bord et menus         | 15 min |
| Démonstration     | Processus principal              | 20 min |
| Exercice pratique | Réalisation par les participants | 30 min |
| Correction        | Analyse des erreurs              | 15 min |
| Deuxième exercice | Processus complet                | 25 min |
| Évaluation        | Tâche individuelle               | 15 min |
| Questions         | Questions et prochaines étapes   | 15 min |

Le plan doit réserver plus de temps à la pratique qu’à la présentation.

---

# Former aux erreurs fréquentes

Les utilisateurs doivent aussi apprendre à reconnaître les erreurs.

Par exemple :

- sélectionner la mauvaise année scolaire ;
- choisir la mauvaise classe ;
- créer un doublon ;
- attribuer un mauvais rôle ;
- enregistrer une note dans la mauvaise évaluation ;
- enregistrer un paiement pour le mauvais élève ;
- partager ses identifiants ;
- quitter une page sans enregistrer ;
- confondre brouillon et validation ;
- modifier une donnée sans autorisation.

Une formation qui montre uniquement le parcours idéal laisse l’utilisateur sans repère lorsqu’une difficulté apparaît.

---

# Former à la vérification

Chaque action importante doit inclure une étape de contrôle.

## Exemple

Après avoir enregistré une présence, l’utilisateur doit vérifier :

- la bonne date ;
- la bonne classe ;
- le nombre d’élèves ;
- les statuts enregistrés ;
- la confirmation de sauvegarde.

Après avoir saisi des notes, il doit vérifier :

- la bonne évaluation ;
- la bonne classe ;
- l’échelle utilisée ;
- les absences ;
- les valeurs inhabituelles ;
- la sauvegarde.

> **Une tâche n’est pas terminée lorsque l’utilisateur clique sur Enregistrer. Elle est terminée lorsqu’il vérifie le résultat.**

---

# Former à la sécurité

Chaque formation doit rappeler les règles essentielles :

- ne pas partager son mot de passe ;
- ne pas utiliser le compte d’un collègue ;
- fermer sa session sur un appareil partagé ;
- vérifier l’identité avant de modifier un dossier ;
- ne pas envoyer de données sensibles dans un groupe public ;
- signaler rapidement tout accès suspect ;
- respecter les limites de son rôle ;
- ne pas contourner une permission.

La facilité d’utilisation ne doit jamais affaiblir la protection des données.

---

# Évaluer les participants

Les méthodes d’évaluation peuvent inclure :

- observation directe ;
- exercice pratique ;
- scénario complet ;
- questions orales ;
- quiz ;
- checklist de compétences ;
- utilisation réelle durant les premiers jours.

## Exemple de validation

| Compétence                      | Réussi | À revoir |
| ------------------------------- | ------ | -------- |
| Se connecter                    |        |          |
| Retrouver sa classe             |        |          |
| Réaliser l’action principale    |        |          |
| Corriger une erreur simple      |        |          |
| Vérifier le résultat            |        |          |
| Demander de l’aide correctement |        |          |

Un utilisateur qui échoue à une étape essentielle doit recevoir un accompagnement complémentaire.

---

# Gérer les absences

Les utilisateurs absents ne doivent pas être considérés comme formés.

Il faut prévoir :

- session de rattrapage ;
- accompagnement individuel ;
- autoformation suivie d’une évaluation ;
- formation par un ambassadeur qualifié ;
- report de certaines responsabilités jusqu’à validation.

Un utilisateur non formé ne doit pas recevoir immédiatement une tâche critique sans accompagnement.

---

# Les ambassadeurs de formation

Les ambassadeurs internes peuvent aider à :

- accueillir les nouveaux utilisateurs ;
- rappeler les procédures ;
- répondre aux questions simples ;
- observer les difficultés ;
- organiser de petits rappels ;
- transmettre les besoins à RELAIS.

Ils doivent recevoir une formation plus approfondie sur :

- les processus principaux ;
- les erreurs fréquentes ;
- les limites de leur rôle ;
- le processus d’escalade ;
- l’utilisation de la documentation ;
- la protection des données.

---

# La documentation de formation

Chaque support doit être :

- clair ;
- court ;
- structuré ;
- adapté au rôle ;
- illustré lorsque nécessaire ;
- cohérent avec la version actuelle de KARMDA ;
- facile à retrouver ;
- testé avec de vrais utilisateurs.

Les documents doivent expliquer :

1. l’objectif ;
2. les prérequis ;
3. les étapes ;
4. le résultat attendu ;
5. les erreurs fréquentes ;
6. la méthode de vérification ;
7. la procédure d’assistance.

# Définition d’une formation réussie

Une formation KARMDA est réussie lorsque :

- les participants comprennent l’objectif de leurs tâches ;
- ils peuvent accéder à leur compte ;
- ils réalisent les opérations essentielles ;
- ils savent vérifier leurs résultats ;
- ils reconnaissent les erreurs fréquentes ;
- ils respectent les règles de sécurité ;
- ils savent utiliser la documentation ;
- ils savent demander de l’aide ;
- ils peuvent reproduire les actions sans dépendre constamment du formateur.

---

# Ce qu’il faut retenir

Retenez les principes suivants :

- la formation doit suivre le travail de l’utilisateur ;
- chaque rôle nécessite un contenu adapté ;
- la démonstration doit être suivie de pratique ;
- la présence ne prouve pas la compétence ;
- l’utilisateur doit savoir vérifier son travail ;
- les erreurs fréquentes doivent être enseignées ;
- la sécurité fait partie de la formation ;
- les supports doivent rester disponibles après la session ;
- les difficultés observées doivent produire des rappels ciblés ;
- le but final est l’autonomie.

---

## Phrase à retenir

> **« Montrer une fonctionnalité informe l’utilisateur. Lui faire accomplir son travail dans KARMDA le forme réellement. »**

7. Pourquoi les erreurs fréquentes doivent-elles être enseignées ?

10. À quelles conditions une formation peut-elle être considérée comme réussie ?

## Question de réflexion

Un formateur vient de terminer une présentation de trois heures devant quarante enseignants.

La démonstration s’est bien déroulée, les participants ont applaudi et personne n’a posé de question.

Cependant, aucun enseignant n’a manipulé KARMDA pendant la session.

Peut-on considérer les enseignants comme formés ?

Expliquez ce qui manque, les risques pour la mise en production et la manière dont vous réorganiseriez la formation afin de vérifier la capacité réelle des participants.
