# KARMDA

# 01 - Méthode de formation

## Formation RELAIS

> Ce document présente la méthode officielle utilisée par RELAIS pour préparer, conduire, évaluer et améliorer les formations KARMDA.

**Objectif**

À la fin de cette lecture, vous serez capable de concevoir une session de formation adaptée aux utilisateurs, de la conduire avec méthode, de vérifier les compétences acquises et d’organiser le suivi nécessaire.

**Temps estimé**

30 minutes

**Prérequis**

- README.md du dossier Formation ;
- connaissance générale de KARMDA ;
- compréhension des rôles et responsabilités des utilisateurs ;
- connaissance du périmètre du déploiement concerné.

---

# Pourquoi une méthode de formation est nécessaire

Deux formateurs peuvent connaître parfaitement KARMDA et produire des résultats très différents.

Le premier peut :

- présenter rapidement toutes les fonctionnalités ;
- parler pendant plusieurs heures ;
- répondre à quelques questions ;
- terminer lorsque le temps est écoulé.

Le second peut :

- identifier les tâches essentielles ;
- adapter le contenu aux participants ;
- expliquer les processus ;
- faire pratiquer chaque utilisateur ;
- observer les difficultés ;
- évaluer les compétences ;
- organiser un suivi.

Les deux auront présenté KARMDA.

Mais seul le second aura réellement formé les utilisateurs.

Une méthode commune permet à RELAIS de garantir une qualité de formation cohérente, quel que soit :

- le formateur ;
- l’établissement ;
- le nombre de participants ;
- le lieu ;
- le rôle concerné ;
- le niveau d’expérience numérique.

---

# Notre philosophie

Chez RELAIS, une formation ne se mesure pas au nombre de pages présentées ni à la durée de la session.

Elle se mesure à ce que l’utilisateur peut accomplir ensuite.

> **Le formateur n’a pas réussi parce qu’il a tout expliqué. Il a réussi lorsque l’utilisateur peut agir correctement sans lui.**

La méthode RELAIS repose sur quatre résultats :

1. l’utilisateur comprend le sens de son travail ;
2. il sait réaliser les actions essentielles ;
3. il sait vérifier le résultat ;
4. il sait réagir lorsqu’une difficulté apparaît.

---

# Le rôle du formateur RELAIS

Le formateur n’est pas uniquement une personne qui connaît le logiciel.

Il doit être capable de :

- comprendre le travail des utilisateurs ;
- expliquer simplement ;
- observer les réactions ;
- adapter son rythme ;
- créer un environnement rassurant ;
- détecter les incompréhensions ;
- corriger sans humilier ;
- évaluer objectivement ;
- documenter les difficultés ;
- améliorer les futurs supports.

Le formateur représente également RELAIS.

Son comportement influence directement la confiance de l’établissement envers KARMDA.

---

# Les responsabilités du formateur

Avant la session, le formateur doit :

- comprendre le contexte de l’établissement ;
- identifier les participants ;
- connaître leurs rôles ;
- confirmer les objectifs ;
- préparer les comptes ;
- tester l’environnement ;
- organiser les exercices ;
- vérifier le matériel ;
- préparer les supports ;
- anticiper les risques.

Pendant la session, il doit :

- expliquer les objectifs ;
- créer un climat de confiance ;
- utiliser un langage simple ;
- respecter le rythme du groupe ;
- démontrer les processus ;
- faire pratiquer les participants ;
- observer les difficultés ;
- corriger les erreurs ;
- vérifier la compréhension ;
- gérer le temps.

Après la session, il doit :

- évaluer les compétences ;
- enregistrer les présences ;
- transmettre les supports ;
- documenter les difficultés ;
- signaler les besoins de rattrapage ;
- répondre aux questions restantes ;
- produire un compte rendu ;
- améliorer la documentation.

---

# La méthode RELAIS en huit étapes

La formation KARMDA suit huit étapes.

```text
1. Analyser
    ↓
2. Définir
    ↓
3. Préparer
    ↓
4. Accueillir
    ↓
5. Expliquer
    ↓
6. Faire pratiquer
    ↓
7. Évaluer
    ↓
8. Suivre
```

Chaque étape protège la suivante.

Une mauvaise analyse produit un mauvais contenu.

Une mauvaise préparation produit une session confuse.

Une formation sans pratique produit une illusion de compétence.

Une formation sans évaluation laisse les difficultés invisibles.

---

# 1. Analyser le besoin

Avant de préparer les diapositives ou les exercices, le formateur doit comprendre le besoin réel.

## Questions à poser

- Qui seront les participants ?
- Quels rôles occupent-ils ?
- Quelles tâches doivent-ils réaliser dans KARMDA ?
- Quelles fonctionnalités sont incluses dans le déploiement ?
- Quelles fonctionnalités ne sont pas encore activées ?
- Quel est leur niveau d’expérience numérique ?
- Utiliseront-ils un téléphone, une tablette ou un ordinateur ?
- La formation se fera-t-elle en français ou dans une autre langue ?
- Quels processus utilisent-ils aujourd’hui ?
- Quelles difficultés ont déjà été identifiées ?
- Quand commenceront-ils à utiliser KARMDA ?
- Quelles tâches sont critiques dès le premier jour ?

## Exemple

Une formation destinée à des enseignants qui utiliseront uniquement les présences ne doit pas consacrer du temps à :

- la création des comptes ;
- la configuration des années scolaires ;
- la gestion financière ;
- l’administration des permissions.

Le contenu doit suivre leurs responsabilités réelles.

---

# Identifier les profils d’apprentissage

Tous les participants ne possèdent pas le même niveau.

Il peut être utile de distinguer :

## Débutant numérique

La personne peut avoir besoin d’aide pour :

- utiliser un navigateur ;
- saisir une adresse ;
- gérer un mot de passe ;
- distinguer un bouton d’un champ ;
- comprendre les messages de confirmation.

## Utilisateur intermédiaire

La personne sait utiliser les outils courants, mais doit apprendre les processus KARMDA.

## Utilisateur avancé

La personne comprend rapidement l’interface et peut éventuellement devenir ambassadeur interne.

Cette classification ne doit jamais être utilisée pour étiqueter ou humilier les participants.

Elle sert uniquement à adapter :

- le rythme ;
- les groupes ;
- les exercices ;
- l’accompagnement.

---

# 2. Définir les objectifs pédagogiques

Une session doit posséder des objectifs précis.

## Mauvais objectif

> Présenter le module de présence.

Cet objectif décrit l’action du formateur.

Il ne précise pas ce que l’utilisateur saura faire.

## Bon objectif

> À la fin de la session, chaque enseignant sera capable de retrouver sa classe, d’ouvrir une session de présence, d’enregistrer les statuts des élèves, de vérifier la sauvegarde et de corriger une erreur autorisée.

Un objectif pédagogique doit décrire une action observable.

## Verbes utiles

- identifier ;
- retrouver ;
- créer ;
- enregistrer ;
- modifier ;
- vérifier ;
- corriger ;
- consulter ;
- expliquer ;
- signaler ;
- démontrer.

Éviter les objectifs vagues comme :

- connaître ;
- comprendre globalement ;
- être familiarisé ;
- avoir vu.

Ces éléments peuvent être utiles, mais ils sont difficiles à évaluer seuls.

---

# Classer les compétences

Les compétences peuvent être réparties en trois niveaux.

## Compétences essentielles

Elles sont indispensables pour travailler dès le lancement.

Exemples :

- se connecter ;
- retrouver sa classe ;
- enregistrer une présence ;
- saisir une note ;
- consulter un dossier ;
- enregistrer un paiement.

## Compétences importantes

Elles améliorent l’autonomie, mais peuvent être apprises après les premières tâches.

Exemples :

- utiliser des filtres ;
- corriger une erreur ;
- consulter un rapport ;
- gérer une situation exceptionnelle.

## Compétences avancées

Elles concernent des usages moins fréquents ou des responsabilités particulières.

Exemples :

- configurer une règle ;
- exporter des données ;
- analyser plusieurs indicateurs ;
- gérer des permissions spécifiques.

La formation initiale doit privilégier les compétences essentielles.

# Préparer les comptes

Avant la session, vérifier que :

- chaque participant possède un compte ;
- le compte correspond au bon établissement ;
- le rôle est correct ;
- les permissions sont correctes ;
- les mots de passe temporaires fonctionnent ;
- le processus de première connexion fonctionne ;
- les comptes de démonstration sont clairement identifiés ;
- les données utilisées pour les exercices sont disponibles.

Lorsque les comptes personnels ne peuvent pas être créés à temps, une solution temporaire doit être organisée sans compromettre la sécurité.

Les participants ne doivent pas partager un même compte personnel.

---

# Préparer l’environnement

Le formateur doit tester :

- l’adresse de KARMDA ;
- la connexion Internet ;
- le vidéoprojecteur ;
- l’écran ;
- le son, si nécessaire ;
- les prises électriques ;
- les rallonges ;
- les appareils ;
- les navigateurs ;
- les liens ;
- les fichiers ;
- les supports ;
- les comptes ;
- les principales actions de la démonstration.

Il est recommandé de réaliser un test complet dans les mêmes conditions que la formation.

---

# Préparer une solution de secours

Une difficulté technique peut toujours apparaître.

Le formateur doit prévoir :

- une deuxième connexion Internet ;
- des captures d’écran ;
- une version PDF des supports ;
- des fiches imprimées ;
- des exercices hors ligne ;
- un compte de secours autorisé ;
- une méthode pour poursuivre la partie théorique ;
- une nouvelle date si la pratique devient impossible.

La solution de secours doit permettre de préserver la valeur de la session.

Elle ne doit pas conduire à considérer les utilisateurs comme formés s’ils n’ont pas pu pratiquer.

---

# Préparer les scénarios

Les exercices doivent représenter des situations réelles.

## Exemple pour un enseignant

> Vous enseignez dans la classe 6e A. À 8 h, vous devez enregistrer la présence. Deux élèves sont absents, un élève arrive en retard et un autre possède une absence justifiée. Enregistrez les statuts puis vérifiez que la session est correctement sauvegardée.

## Exemple pour un administrateur

> Un nouvel élève doit être inscrit dans la classe 5e B. Retrouvez d’abord son dossier afin d’éviter un doublon, complétez les informations obligatoires, créez l’inscription et vérifiez qu’il apparaît dans la bonne classe.

Un bon scénario comprend :

- un contexte ;
- un objectif ;
- les informations nécessaires ;
- une action ;
- un résultat attendu ;
- une méthode de vérification.

---

# Préparer les données d’exercice

Les données utilisées doivent être :

- réalistes ;
- faciles à reconnaître ;
- adaptées au rôle ;
- sans risque pour les données officielles ;
- clairement identifiées comme données de test lorsque nécessaire.

Éviter d’utiliser des données fictives qui ressemblent trop aux dossiers réels sans signalement clair.

Après la formation, les données de test doivent être supprimées ou archivées selon le processus prévu.

---

# Préparer le plan de session

Chaque session doit suivre une structure claire.

## Modèle recommandé

| Étape           | Objectif                               | Durée indicative |
| --------------- | -------------------------------------- | ---------------: |
| Accueil         | Installer le cadre et les objectifs    |           10 min |
| Contexte        | Expliquer le rôle de KARMDA            |           10 min |
| Connexion       | Accéder au compte et sécuriser l’accès |           15 min |
| Démonstration 1 | Montrer la tâche principale            |           15 min |
| Pratique 1      | Faire reproduire la tâche              |           25 min |
| Correction      | Traiter les erreurs observées          |           10 min |
| Démonstration 2 | Présenter un processus complémentaire  |           15 min |
| Pratique 2      | Faire réaliser un scénario complet     |           30 min |
| Évaluation      | Vérifier la compétence individuelle    |           20 min |
| Questions       | Répondre aux difficultés restantes     |           15 min |
| Conclusion      | Résumer et expliquer le support        |           10 min |

Le plan doit conserver une certaine souplesse.

Cependant, le formateur ne doit pas sacrifier la pratique pour terminer toutes ses diapositives.

---

# Préparer les supports

Les supports peuvent inclure :

- présentation ;
- guide utilisateur ;
- fiche pratique ;
- checklist ;
- vidéo ;
- exercice ;
- questionnaire ;
- feuille de présence ;
- grille d’évaluation ;
- formulaire de satisfaction.

Chaque support doit correspondre à la version actuelle de KARMDA.

Les captures, noms de menus et procédures obsolètes doivent être retirés.

---

# 4. Accueillir les participants

Les premières minutes influencent toute la session.

Le formateur doit créer un environnement où les participants se sentent capables d’apprendre.

## Au début de la session

- accueillir les participants ;
- se présenter ;
- présenter RELAIS ;
- rappeler l’objectif du projet ;
- expliquer le programme ;
- préciser la durée ;
- indiquer les pauses ;
- encourager les questions ;
- rappeler que les erreurs font partie de l’apprentissage ;
- vérifier les appareils ;
- confirmer que les comptes fonctionnent.

## Formulation possible

> Aujourd’hui, l’objectif n’est pas de mémoriser toute la plateforme. Nous allons nous concentrer sur les actions que vous devrez réellement réaliser. Vous aurez le temps de pratiquer, de faire des erreurs et de recommencer. À la fin, chacun devra pouvoir effectuer les tâches essentielles avec son propre compte.

---

# Établir les règles de session

Les règles peuvent inclure :

- poser les questions au moment où elles apparaissent ;
- éviter de manipuler des données officielles hors exercice ;
- ne pas partager les mots de passe ;
- suivre les étapes avant d’explorer seul ;
- demander de l’aide plutôt que rester bloqué ;
- respecter le rythme du groupe ;
- limiter les conversations parallèles ;
- laisser chacun pratiquer.

Ces règles doivent rester simples et respectueuses.

---

# Observer le groupe

Dès les premières étapes, le formateur doit observer :

- qui se connecte facilement ;
- qui reste bloqué ;
- qui n’ose pas demander ;
- qui avance trop vite ;
- qui aide les autres ;
- qui semble inquiet ;
- qui ne possède pas l’équipement nécessaire ;
- qui pourrait devenir ambassadeur.

Cette observation aide à adapter la suite de la session.

---

# 5. Expliquer et démontrer

Une démonstration doit être courte, structurée et visible.

## Avant de cliquer

Expliquer :

- la situation ;
- l’objectif ;
- le rôle responsable ;
- les informations nécessaires ;
- le résultat attendu.

## Pendant la démonstration

Le formateur doit :

- avancer lentement ;
- annoncer chaque étape ;
- expliquer les choix importants ;
- éviter les raccourcis non expliqués ;
- montrer les confirmations ;
- signaler les erreurs fréquentes ;
- vérifier que les participants suivent.

## Après la démonstration

Résumer :

- ce qui a été réalisé ;
- les étapes principales ;
- le point de vérification ;
- les erreurs à éviter.

---

# La méthode « Pourquoi, Où, Comment, Vérifier »

Pour chaque tâche, le formateur doit répondre à quatre questions.

## Pourquoi ?

Pourquoi cette action est-elle nécessaire ?

## Où ?

Dans quelle partie de KARMDA faut-il intervenir ?

## Comment ?

Quelles étapes doivent être suivies ?

## Vérifier ?

Comment savoir que l’action a réussi ?

## Exemple

### Pourquoi ?

Enregistrer les présences permet à l’établissement de suivre les absences et les retards.

### Où ?

Dans le menu de présence de la classe concernée.

### Comment ?

Sélectionner la date, ouvrir la session, attribuer les statuts puis enregistrer.

### Vérifier ?

Contrôler la date, la classe, les statuts et le message de confirmation.

Cette méthode évite une formation basée uniquement sur des clics mémorisés.

---

# Limiter le jargon

Le formateur doit utiliser le vocabulaire de l’établissement.

Éviter autant que possible :

- endpoint ;
- synchronisation asynchrone ;
- rôle RBAC ;
- environnement de production ;
- contrainte de base de données ;
- identifiant technique ;
- erreur serveur 500.

Préférer :

- accès ;
- autorisation ;
- compte ;
- donnée enregistrée ;
- problème technique ;
- information manquante ;
- service temporairement indisponible.

Le jargon technique peut être expliqué lorsqu’il est réellement utile.

---

# Ne pas tout montrer

Un formateur expérimenté peut être tenté de montrer toutes les possibilités.

Cela crée souvent :

- surcharge cognitive ;
- confusion ;
- questions hors périmètre ;
- perte de temps ;
- mémorisation faible.

Le contenu doit être limité aux responsabilités du rôle et à la phase actuelle.

Une fonctionnalité non activée ne doit pas être présentée comme immédiatement disponible.

---

# 6. Faire pratiquer

La pratique est le cœur de la formation.

Après chaque démonstration importante, les participants doivent réaliser l’action.

## Le formateur doit demander

> Prenez maintenant votre propre appareil et reproduisez cette opération.

Il ne doit pas demander uniquement :

> Est-ce que tout le monde a compris ?

Une réponse positive collective ne constitue pas une preuve.

---

# Niveaux de pratique

## Pratique guidée

Le formateur donne les étapes une par une.

Adaptée au premier exercice.

## Pratique semi-guidée

Le formateur donne le scénario, mais rappelle certains repères.

Adaptée au deuxième exercice.

## Pratique autonome

Le participant reçoit uniquement l’objectif et doit réaliser le processus.

Adaptée à l’évaluation.

La progression recommandée est :

```text
Je montre
    ↓
Nous faisons ensemble
    ↓
Vous faites avec aide
    ↓
Vous faites seul
```

---

# Observer sans prendre le contrôle

Lorsqu’un participant rencontre une difficulté, le formateur ne doit pas immédiatement prendre son appareil.

Il peut demander :

- Qu’essayez-vous de faire ?
- Quelle étape avez-vous déjà réalisée ?
- Que voyez-vous à l’écran ?
- Où pensez-vous pouvoir trouver cette option ?
- Quel résultat attendez-vous ?

Cette méthode aide le participant à construire son raisonnement.

Le formateur peut intervenir directement lorsque :

- une donnée sensible risque d’être modifiée ;
- une erreur importante peut être commise ;
- le groupe entier est bloqué ;
- le temps devient critique.

---

# Corriger une erreur

Une correction efficace suit cette séquence :

```text
Observer
    ↓
Demander à l’utilisateur d’expliquer
    ↓
Identifier l’étape incorrecte
    ↓
Montrer ou rappeler le principe
    ↓
Faire recommencer
    ↓
Vérifier
```

Le formateur doit corriger l’action sans attaquer la personne.

## À éviter

> Non, ce n’est pas du tout comme ça.

## À préférer

> Vous êtes dans la bonne classe, mais regardons la date sélectionnée avant d’enregistrer. Quelle date devez-vous utiliser pour cette session ?

---

# Gérer les différents rythmes

## Participants rapides

Ils peuvent :

- réaliser un exercice complémentaire ;
- vérifier leur résultat ;
- aider un collègue avec l’accord du formateur ;
- explorer une fonction autorisée.

Ils ne doivent pas imposer leur rythme au groupe.

## Participants en difficulté

Ils peuvent recevoir :

- une explication différente ;
- une fiche simplifiée ;
- une assistance individuelle ;
- plus de temps ;
- un exercice réduit ;
- une session de rattrapage.

Le but n’est pas que tout le monde termine exactement au même moment.

Le but est que chacun maîtrise les compétences essentielles.

---

# Gérer les questions

Les questions peuvent être classées.

## Question liée au contenu

Répondre immédiatement lorsqu’elle aide le groupe.

## Question individuelle

Répondre brièvement ou traiter pendant la pratique.

## Question hors périmètre

La noter et y revenir à la fin ou après la session.

## Demande d’évolution

La documenter sans promettre une réalisation.

## Problème technique

Le signaler et utiliser la solution de secours si nécessaire.

Le formateur peut utiliser un « parking des questions » :

| Question | Catégorie | Réponse prévue |
| -------- | --------- | -------------- |
|          |           |                |
|          |           |                |

Cela évite de perdre les questions sans interrompre toute la progression.

---

# 7. Évaluer les compétences

L’évaluation doit porter sur les tâches essentielles.

Elle ne doit pas servir à piéger ou humilier.

Elle permet de savoir :

- qui est prêt ;
- qui nécessite un rappel ;
- quelles étapes restent mal comprises ;
- si la formation doit être adaptée ;
- si le lancement présente un risque.

---

# Types d’évaluation

## Observation directe

Le formateur observe l’utilisateur pendant une tâche.

## Exercice autonome

Le participant réalise un scénario sans aide.

## Questions de compréhension

Le participant explique :

- pourquoi il réalise l’action ;
- comment il vérifie le résultat ;
- que faire en cas d’erreur.

## Quiz

Utile pour vérifier certaines règles :

- sécurité ;
- responsabilités ;
- processus ;
- erreurs fréquentes.

## Utilisation réelle

Les premiers jours permettent de confirmer que la compétence est réellement appliquée.

---

# Grille d’évaluation

| Compétence                        | Réussi seul | Réussi avec aide | À revoir | Non évalué |
| --------------------------------- | ----------- | ---------------- | -------- | ---------- |
| Se connecter                      |             |                  |          |            |
| Accéder à la bonne section        |             |                  |          |            |
| Réaliser la tâche principale      |             |                  |          |            |
| Vérifier le résultat              |             |                  |          |            |
| Corriger une erreur simple        |             |                  |          |            |
| Respecter les règles de sécurité  |             |                  |          |            |
| Signaler correctement un problème |             |                  |          |            |

---

# Critères de validation

Un participant peut être considéré comme opérationnel lorsqu’il :

- se connecte avec son propre compte ;
- retrouve l’espace qui correspond à son rôle ;
- réalise la tâche essentielle ;
- utilise les bonnes données ;
- vérifie le résultat ;
- évite les erreurs critiques ;
- sait demander de l’aide ;
- peut reproduire l’action sans guidage permanent.

Une réussite obtenue uniquement avec l’intervention continue du formateur indique qu’un accompagnement supplémentaire est nécessaire.

---

# Ne pas confondre mémoire et compréhension

Un participant peut reproduire immédiatement les clics observés sans comprendre le processus.

Pour vérifier sa compréhension, modifier légèrement le scénario.

## Exemple

Pendant la démonstration, la présence concernait la classe 6e A.

Pendant l’évaluation, demander :

> Enregistrez maintenant la présence de la 5e B à la date d’hier et vérifiez le nombre d’élèves absents.

Le participant doit adapter son action.

---

# Documenter les résultats

Après l’évaluation, le formateur doit indiquer :

- les compétences validées ;
- les difficultés observées ;
- les participants nécessitant un rappel ;
- les risques éventuels ;
- les recommandations ;
- la date du prochain suivi.

Les résultats individuels doivent être traités avec respect et confidentialité.

---

# 8. Organiser le suivi

Une seule session ne suffit pas toujours.

Le suivi doit être planifié en fonction des résultats.

## Possibilités

- session de rattrapage ;
- rappel ciblé ;
- accompagnement individuel ;
- fiche pratique ;
- vidéo courte ;
- exercice complémentaire ;
- observation pendant les premiers jours ;
- soutien par un ambassadeur ;
- nouvelle évaluation.

---

# Déterminer les priorités de suivi

Le suivi doit être prioritaire lorsque :

- un utilisateur réalise une tâche critique ;
- plusieurs participants commettent la même erreur ;
- une règle de sécurité est mal comprise ;
- une procédure produit des données incorrectes ;
- un groupe n’a pas pu pratiquer ;
- un utilisateur clé était absent ;
- la session a été interrompue ;
- les participants dépendent encore entièrement du formateur.

---

# Produire le compte rendu de formation

Le compte rendu doit inclure :

- l’établissement ;
- la date ;
- le public ;
- les formateurs ;
- les objectifs ;
- les participants présents ;
- les participants absents ;
- le contenu traité ;
- les exercices réalisés ;
- les compétences évaluées ;
- les difficultés observées ;
- les questions ouvertes ;
- les besoins de rattrapage ;
- les actions suivantes.

## Modèle

```text
Établissement :

Date :

Public :

Formateur :

Participants attendus :

Participants présents :

Objectifs de la session :

Contenu traité :

Exercices réalisés :

Compétences validées :

Difficultés principales :

Participants nécessitant un suivi :

Questions ouvertes :

Actions à réaliser :

Prochaine étape :
```

# Recueillir les retours

Le retour des participants aide à améliorer la formation.

Questions utiles :

- Le contenu correspondait-il à votre travail ?
- Avez-vous eu suffisamment de temps pour pratiquer ?
- Quelle étape reste difficile ?
- Quel support vous serait le plus utile ?
- Vous sentez-vous capable de reproduire les tâches ?
- Qu’aurions-nous pu expliquer autrement ?

La satisfaction seule ne suffit pas.

Une session peut être agréable mais inefficace.

Les retours doivent être comparés aux résultats des exercices.

---

# Améliorer la méthode

Après chaque formation, le formateur doit analyser :

- les étapes qui ont pris trop de temps ;
- les explications qui ont créé de la confusion ;
- les questions récurrentes ;
- les erreurs les plus fréquentes ;
- les exercices trop simples ou trop difficiles ;
- les problèmes techniques ;
- la pertinence des supports ;
- la durée réelle ;
- la taille du groupe ;
- les compétences non maîtrisées.

Les améliorations doivent être intégrées aux documents et supports.

---

# Former en présentiel

En présentiel, vérifier particulièrement :

- la visibilité de l’écran ;
- la disposition de la salle ;
- les appareils disponibles ;
- les prises ;
- le bruit ;
- la connexion ;
- la possibilité de circuler entre les participants ;
- la présence d’un assistant pour les grands groupes.

Le formateur doit éviter de rester immobile devant l’écran pendant toute la session.

Il doit observer les appareils et les exercices.

---

# Former à distance

À distance, vérifier :

- le lien de réunion ;
- la connexion ;
- le son ;
- le partage d’écran ;
- les droits d’accès ;
- le canal de questions ;
- la capacité des participants à partager leur écran ;
- les supports transmis à l’avance.

Les sessions à distance doivent souvent être plus courtes.

Il est recommandé de :

- limiter le nombre de participants ;
- faire des pauses ;
- appeler régulièrement les participants par leur nom ;
- demander des démonstrations individuelles ;
- vérifier que les caméras ou écrans masqués ne cachent pas un décrochage complet.

---

# Former un grand groupe

Une présentation devant un grand groupe peut servir à :

- présenter le projet ;
- expliquer les bénéfices ;
- montrer les principaux processus ;
- répondre aux questions générales.

Elle ne suffit pas pour valider les compétences.

Le grand groupe doit être complété par :

- des ateliers ;
- des petits groupes ;
- des exercices ;
- des évaluations ;
- des ambassadeurs ;
- des sessions de rattrapage.

---

# Former des utilisateurs peu à l’aise avec le numérique

Le formateur peut :

- ralentir le rythme ;
- utiliser des mots simples ;
- éviter plusieurs nouvelles notions en même temps ;
- répéter les étapes ;
- laisser une fiche imprimée ;
- utiliser les mêmes repères visuels ;
- vérifier les gestes de base ;
- proposer un accompagnement individuel ;
- valoriser chaque progrès.

Il ne doit pas supposer qu’une difficulté numérique signifie un manque d’intelligence ou de motivation.

---

# Former dans plusieurs langues

Lorsque certains participants maîtrisent mieux une langue locale :

- identifier cette réalité avant la session ;
- utiliser un interprète compétent si nécessaire ;
- préparer des supports simples ;
- éviter les traductions improvisées de termes importants ;
- vérifier la compréhension individuellement ;
- utiliser des démonstrations visuelles ;
- faire reformuler les étapes.

Une traduction littérale ne suffit pas toujours.

Le sens du processus doit rester correct.

---

# Gérer les situations difficiles

## Participant qui monopolise la session

Remercier la personne, répondre brièvement et proposer de poursuivre après la session si la question est individuelle.

## Participant qui refuse d’utiliser KARMDA

Écouter la raison, reformuler la préoccupation et relier la formation à son travail réel.

Le formateur ne doit pas transformer la session en confrontation.

## Direction qui interrompt constamment

Rappeler respectueusement le plan et proposer un temps spécifique pour les décisions de direction.

## Groupe silencieux

Poser des questions concrètes, organiser des exercices et demander des reformulations individuelles.

## Participants sans appareil

Organiser des binômes temporairement, mais prévoir une pratique individuelle ultérieure.

## Connexion indisponible

Utiliser les supports de secours, expliquer les processus et reporter formellement la validation pratique.

---

# Erreurs fréquentes du formateur

## Parler trop longtemps

Une longue explication réduit le temps de pratique.

---

## Suivre les menus plutôt que le travail

Les utilisateurs retiennent les tâches, pas l’architecture interne du produit.

---

## Aller trop vite

Le formateur connaît déjà le parcours.

Les participants le découvrent.

---

## Poser uniquement « Vous avez compris ? »

Cette question produit souvent un oui poli.

Demander une action ou une reformulation.

---

## Prendre l’appareil du participant

Cela résout le problème du formateur, pas celui de l’apprenant.

---

## Montrer trop de fonctionnalités

La surcharge réduit la mémorisation.

---

## Ignorer les erreurs

Une erreur non corrigée peut devenir une habitude.

---

## Corriger publiquement de manière humiliante

La peur réduit les questions et cache les difficultés.

---

## Considérer les absents comme formés

La liste d’invitation ne remplace pas la présence et la pratique.

---

## Terminer sans évaluation

Sans évaluation, RELAIS ne sait pas si les utilisateurs sont prêts.

---

## Promettre des évolutions

Le formateur doit documenter les demandes, pas engager le produit sans validation.

---

# Bonnes pratiques RELAIS

✔ Analyser les rôles avant de créer le contenu.

✔ Définir des objectifs observables.

✔ Prioriser les tâches essentielles.

✔ Tester les comptes et le matériel.

✔ Préparer des scénarios réalistes.

✔ Prévoir une solution de secours.

✔ Expliquer le pourquoi avant les étapes.

✔ Démontrer lentement.

✔ Faire pratiquer chaque participant.

✔ Observer sans prendre immédiatement le contrôle.

✔ Corriger avec respect.

✔ Adapter le rythme.

✔ Évaluer les compétences individuellement.

✔ Documenter les résultats.

✔ Organiser les rattrapages.

✔ Recueillir les retours.

✔ Améliorer les supports après chaque session.

# Livrables de la formation

Pour chaque session, le dossier doit contenir :

- l’analyse du besoin ;
- la liste des participants ;
- les objectifs pédagogiques ;
- le plan de session ;
- les supports ;
- les exercices ;
- les comptes ou données de test utilisés ;
- la feuille de présence ;
- la grille d’évaluation ;
- les résultats ;
- les questions ouvertes ;
- le compte rendu ;
- le plan de suivi ;
- les retours des participants ;
- les améliorations proposées.

---

# Conditions de réussite

Une session peut être considérée comme réussie lorsque :

- le bon public a été formé ;
- les objectifs correspondent aux responsabilités ;
- les comptes et outils ont fonctionné ;
- les participants ont pratiqué ;
- les compétences essentielles ont été évaluées ;
- les principales erreurs ont été corrigées ;
- les absents sont identifiés ;
- les besoins de suivi sont documentés ;
- les supports ont été transmis ;
- les utilisateurs savent comment demander de l’aide.

---

# Ce qu’il faut retenir

À l’issue de ce chapitre, retenez les idées suivantes :

- une bonne connaissance de KARMDA ne suffit pas pour être un bon formateur ;
- la formation commence par l’analyse du travail réel ;
- les objectifs doivent décrire ce que l’utilisateur saura accomplir ;
- la préparation protège la qualité de la session ;
- la démonstration doit toujours être suivie de pratique ;
- le formateur doit observer avant d’intervenir ;
- les erreurs doivent être corrigées avec respect ;
- la compétence doit être évaluée par une action ;
- les résultats doivent produire un suivi ;
- chaque formation doit améliorer la suivante.

---

## Phrase à retenir

> **« Une formation n’est pas une performance du formateur. C’est une transformation observable de l’utilisateur. »**

## Question de réflexion

Vous devez former vingt enseignants pendant deux heures.

Au début de la session, vous découvrez que :

- six enseignants n’ont pas de compte ;
- cinq n’ont pas apporté d’appareil ;
- la connexion Internet est instable ;
- plusieurs participants doivent quitter la salle après une heure ;
- la direction insiste pour que tous soient déclarés formés le jour même.

Comment réorganisez-vous la session ?

Expliquez :

- ce qui peut être maintenu ;
- ce qui doit être reporté ;
- comment organiser la pratique ;
- comment gérer les participants sans compte ou sans appareil ;
- quelles compétences peuvent réellement être validées ;
- ce que vous communiquez à la direction ;
- quelles actions de suivi deviennent obligatoires.
