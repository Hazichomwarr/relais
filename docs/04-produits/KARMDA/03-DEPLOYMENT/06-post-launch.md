# KARMDA

# 06 - Suivi post-déploiement

## Déploiement RELAIS

> Ce document fait partie du programme officiel de déploiement KARMDA.

**Objectif**

À la fin de cette lecture, vous serez capable d’accompagner un établissement après sa mise en production, de stabiliser son utilisation de KARMDA, de traiter les premières difficultés et de préparer son passage vers le support régulier.

**Temps estimé**

25 minutes

**Prérequis**

- README.md
- 01 - Préparation du déploiement
- 02 - Installation
- 03 - Onboarding de l’établissement
- 04 - Import des données
- 05 - Checklist de mise en production
- formations prévues pour les utilisateurs concernés

---

# Pourquoi ce chapitre est important

La mise en production ne marque pas la fin du déploiement.

Elle marque le début de l’utilisation réelle.

C’est à ce moment que les utilisateurs commencent à travailler dans KARMDA dans leurs conditions quotidiennes :

- avec leurs appareils ;
- avec leur connexion Internet ;
- avec leurs habitudes ;
- avec leurs contraintes ;
- avec leurs responsabilités ;
- avec leurs données réelles ;
- avec la pression de leurs activités habituelles.

Certaines difficultés ne peuvent apparaître qu’après le lancement.

Par exemple :

- un utilisateur oublie son mot de passe ;
- un enseignant ne retrouve pas sa classe ;
- une procédure comprise pendant la formation est mal appliquée ;
- une donnée importée nécessite une correction ;
- un processus réel diffère légèrement de ce qui avait été expliqué ;
- une notification n’atteint pas certains destinataires ;
- un utilisateur revient à l’ancien système par habitude ;
- la direction ne consulte pas encore les nouveaux indicateurs.

Le suivi post-déploiement permet de transformer une mise en production techniquement réussie en adoption durable.

---

# Notre philosophie

Chez RELAIS, nous ne disparaissons pas lorsque KARMDA est lancé.

Nous restons présents jusqu’à ce que l’établissement puisse utiliser la plateforme avec confiance.

Le suivi post-déploiement doit permettre de répondre à trois questions :

1. Le système fonctionne-t-il dans les conditions réelles ?
2. Les utilisateurs l’adoptent-ils dans leur travail quotidien ?
3. L’établissement devient-il progressivement autonome ?

> **Le lancement prouve que KARMDA peut fonctionner. Le suivi prouve que l’établissement peut réussir avec KARMDA.**

---

# Les objectifs du suivi post-déploiement

Cette phase doit permettre de :

- surveiller les premiers jours d’utilisation ;
- détecter rapidement les incidents ;
- aider les utilisateurs à appliquer les processus ;
- corriger les données ou configurations nécessaires ;
- prévenir le retour aux anciennes pratiques ;
- mesurer l’adoption ;
- renforcer les ambassadeurs internes ;
- répondre aux questions récurrentes ;
- documenter les apprentissages ;
- confirmer la stabilité du déploiement ;
- organiser le passage vers le support régulier.

---

# La période de stabilisation

Les premiers jours après la mise en production constituent une période de stabilisation.

Pendant cette période, l’équipe RELAIS doit être plus attentive que durant le fonctionnement habituel.

La durée dépend :

- de la taille de l’établissement ;
- du nombre d’utilisateurs ;
- du nombre de modules activés ;
- de la complexité des processus ;
- de la qualité des données ;
- du niveau d’autonomie des équipes ;
- du nombre de problèmes ouverts au lancement.

Une période indicative peut être organisée ainsi :

```text
Jour du lancement
    ↓
Suivi à 24 heures
    ↓
Suivi à 3 jours
    ↓
Suivi à 7 jours
    ↓
Suivi à 14 jours
    ↓
Bilan à 30 jours
```

Toutes les étapes ne sont pas obligatoires pour chaque projet.

Le niveau d’accompagnement doit être proportionné au risque et aux besoins de l’établissement.

# Définir les responsabilités après le lancement

## Le chef de projet RELAIS

Il :

- coordonne la stabilisation ;
- organise les points de suivi ;
- vérifie l’avancement des corrections ;
- maintient la communication avec l’établissement ;
- distingue les incidents des demandes d’évolution ;
- prépare le bilan de fin de déploiement ;
- organise le passage vers le support.

---

## L’équipe technique RELAIS

Elle :

- analyse les incidents techniques ;
- corrige les défauts confirmés ;
- vérifie les journaux et comportements du système ;
- sécurise les interventions ;
- documente les corrections ;
- informe le chef de projet des risques et délais.

---

## Le formateur ou référent fonctionnel RELAIS

Il :

- répond aux difficultés d’utilisation ;
- vérifie que les processus sont compris ;
- propose des rappels ciblés ;
- identifie les besoins de formation complémentaire ;
- améliore les supports lorsque des incompréhensions se répètent.

---

## Le responsable interne de l’établissement

Il :

- centralise les questions ;
- suit les difficultés des utilisateurs ;
- transmet les informations nécessaires ;
- coordonne les validations ;
- rappelle les nouvelles pratiques ;
- aide à distinguer une erreur d’utilisation d’un incident technique.

---

## Les ambassadeurs internes

Ils :

- accompagnent leurs collègues ;
- répondent aux questions simples ;
- signalent les blocages ;
- encouragent l’utilisation de KARMDA ;
- repèrent les retours aux anciennes méthodes ;
- participent à la montée en autonomie de l’établissement.

---

# Organiser le support renforcé

Avant le lancement, RELAIS doit avoir défini :

- les canaux de contact ;
- les horaires de disponibilité ;
- les personnes autorisées à signaler un incident ;
- les délais de réponse attendus ;
- les niveaux de priorité ;
- le processus d’escalade ;
- le format des informations à fournir.

## Exemple de canal

- WhatsApp ou téléphone pour les incidents urgents ;
- e-mail ou outil de support pour les demandes documentées ;
- réunion de suivi pour les questions non urgentes ;
- base documentaire pour les questions fréquentes.

Un canal rapide ne doit pas supprimer la nécessité de documenter les incidents importants.

| INC-001   |      |             |          |        |          |             |        |
| INC-002   |      |             |          |        |          |             |        |

# Classer les incidents

Tous les problèmes ne possèdent pas le même niveau d’urgence.

## Priorité 1 — Critique

Le problème :

- bloque une opération essentielle ;
- affecte plusieurs utilisateurs ;
- compromet la sécurité ;
- produit des données incorrectes importantes ;
- empêche l’établissement de fonctionner selon le périmètre prévu.

Exemples :

- aucun enseignant ne peut saisir les présences ;
- des utilisateurs voient les données d’un autre établissement ;
- les notes sont calculées incorrectement ;
- les paiements sont attribués aux mauvais élèves ;
- l’application est inaccessible pour tous.

Une priorité critique nécessite :

- une prise en charge immédiate ;
- une communication rapide ;
- une solution temporaire lorsque possible ;
- des mises à jour régulières ;
- une validation après correction.

---

## Priorité 2 — Élevée

Le problème affecte fortement un processus, mais une solution temporaire existe ou le périmètre touché reste limité.

Exemples :

- une classe ne peut pas être utilisée ;
- certains comptes ne fonctionnent pas ;
- un rapport important est incorrect ;
- une notification essentielle échoue pour une partie des utilisateurs.

---

## Priorité 3 — Normale

Le problème gêne l’utilisation sans bloquer les opérations principales.

Exemples :

- un libellé est imprécis ;
- une procédure nécessite une explication ;
- un filtre ne fonctionne pas comme attendu ;
- une information facultative manque.

---

## Priorité 4 — Amélioration

Il ne s’agit pas d’un incident.

La demande concerne :

- une nouvelle fonctionnalité ;
- une préférence d’affichage ;
- une automatisation supplémentaire ;
- une modification hors périmètre ;
- une idée d’amélioration.

Ces demandes doivent être enregistrées séparément et ne pas être présentées comme des anomalies du déploiement.

---

# Distinguer incident, question et évolution

Cette distinction protège la clarté du suivi.

## Incident

Le système ne fonctionne pas comme prévu.

> Un enseignant ne peut pas enregistrer la présence alors qu’il possède le bon rôle.

## Question d’utilisation

Le système fonctionne, mais l’utilisateur ne sait pas encore réaliser l’action.

> Un enseignant demande comment modifier une présence enregistrée.

## Problème de données

Le système fonctionne, mais une information enregistrée est incorrecte ou incomplète.

> Un élève apparaît dans la mauvaise classe.

## Demande d’évolution

L’utilisateur souhaite une capacité qui ne fait pas partie du fonctionnement actuel.

> La direction souhaite recevoir automatiquement un rapport personnalisé chaque soir.

Chaque catégorie doit suivre le bon traitement :

- incident → analyse et correction ;
- question → accompagnement ou formation ;
- donnée → validation et correction ;
- évolution → étude, estimation et décision commerciale ou produit.

---

# Collecter les bonnes informations

Lorsqu’un utilisateur signale un problème, demander des informations précises.

## Informations utiles

- nom de l’utilisateur ;
- rôle ;
- date et heure ;
- appareil utilisé ;
- page concernée ;
- action tentée ;
- résultat attendu ;
- résultat obtenu ;
- message d’erreur ;
- personnes également affectées ;
- capture d’écran lorsque appropriée ;
- niveau d’urgence opérationnelle.

## Mauvais signalement

> KARMDA ne marche pas.

## Signalement exploitable

> Le 22 juillet à 8 h 15, l’enseignant Moussa, connecté avec son compte enseignant sur téléphone Android, a ouvert la classe 5e A. La liste des élèves apparaît, mais le bouton de validation de la présence reste désactivé. Deux autres enseignants de la même classe rencontrent le même problème.

Un bon signalement réduit le temps d’analyse.

---

# Répondre aux incidents

La première réponse ne doit pas attendre la résolution complète.

Elle doit confirmer :

- que le problème a été reçu ;
- son niveau de priorité provisoire ;
- la personne qui le prend en charge ;
- l’heure de la prochaine mise à jour ;
- la solution temporaire disponible, lorsqu’elle existe.

## Exemple

> Nous avons bien reçu le signalement concernant la saisie des présences de la classe 5e A. L’incident est en cours d’analyse avec une priorité élevée. En attendant, veuillez utiliser la feuille temporaire prévue pour la continuité des présences. Une nouvelle mise à jour vous sera transmise avant 10 h 30.

Ne jamais laisser le client sans visibilité pendant l’analyse.

---

# Utiliser des solutions temporaires avec prudence

Une solution temporaire permet de maintenir l’activité pendant la résolution.

Par exemple :

- feuille papier provisoire ;
- fichier sécurisé ;
- saisie différée ;
- utilisation d’un autre compte autorisé ;
- validation manuelle exceptionnelle ;
- report d’une opération non critique.

Toute solution temporaire doit préciser :

- sa durée ;
- les personnes autorisées ;
- les données à conserver ;
- la procédure de réintégration dans KARMDA ;
- le risque de double saisie ;
- la personne chargée du suivi.

Une solution temporaire sans date de fin peut devenir une nouvelle mauvaise habitude.

---

# Suivi du premier jour

Le jour du lancement, RELAIS doit observer les premières opérations importantes.

## Points à vérifier

- les utilisateurs prioritaires se connectent ;
- les rôles et permissions fonctionnent ;
- les premières saisies sont réussies ;
- les données principales sont retrouvées ;
- les utilisateurs savent demander de l’aide ;
- aucun incident critique n’est ouvert ;
- les solutions temporaires éventuelles sont maîtrisées ;
- la direction reçoit un premier retour.

## Bilan de fin de journée

Le bilan doit inclure :

- nombre d’utilisateurs actifs ;
- processus utilisés ;
- incidents signalés ;
- incidents résolus ;
- problèmes restant ouverts ;
- questions récurrentes ;
- actions du lendemain ;
- niveau général de confiance.

---

# Suivi à 24 heures

Le premier suivi formel doit vérifier :

- ce qui a réellement été utilisé ;
- ce qui n’a pas encore été utilisé ;
- les utilisateurs bloqués ;
- les erreurs de données découvertes ;
- les procédures mal comprises ;
- les anciens processus encore utilisés ;
- la satisfaction du responsable interne ;
- les incidents ouverts ;
- les besoins immédiats de rappel ou de correction.

Questions utiles :

- Quelles opérations ont été réalisées avec succès ?
- Quelles opérations n’ont pas pu être réalisées ?
- Quels utilisateurs ont besoin d’aide ?
- Quelles questions reviennent plusieurs fois ?
- Quelles données semblent incorrectes ?
- Existe-t-il un retour à l’ancien système ?
- Quel est le problème le plus important à résoudre aujourd’hui ?

---

# Suivi à 7 jours

Après une semaine, l’analyse doit dépasser les incidents immédiats.

Il faut évaluer :

- l’utilisation réelle des modules ;
- la régularité des opérations ;
- l’autonomie des utilisateurs ;
- l’efficacité des ambassadeurs ;
- la qualité des données produites ;
- le respect des nouveaux processus ;
- les difficultés récurrentes ;
- le niveau de confiance de la direction ;
- l’efficacité du support.

## Exemples d’indicateurs

- pourcentage d’enseignants connectés ;
- nombre de sessions de présence créées ;
- pourcentage de classes ayant enregistré les présences ;
- nombre d’élèves ajoutés ou mis à jour ;
- nombre de notes saisies ;
- nombre de paiements enregistrés ;
- nombre d’incidents ;
- délai moyen de résolution ;
- nombre de questions d’utilisation ;
- nombre d’utilisateurs nécessitant un rappel.

Les indicateurs doivent être interprétés dans le contexte de l’établissement.

Une faible utilisation n’est pas toujours un rejet.

Elle peut être liée au calendrier scolaire ou à l’absence d’une activité prévue cette semaine-là.

---

# Suivi à 30 jours

Le bilan à 30 jours permet de confirmer si le déploiement produit les résultats attendus.

Il doit répondre aux questions suivantes :

1. Les objectifs de la première phase sont-ils atteints ?
2. Les utilisateurs prioritaires utilisent-ils KARMDA régulièrement ?
3. Les processus prévus ont-ils réellement changé ?
4. Les données produites sont-elles fiables ?
5. L’établissement sait-il résoudre les difficultés simples ?
6. Les incidents restants sont-ils maîtrisés ?
7. La direction utilise-t-elle les informations disponibles ?
8. L’établissement peut-il passer au support régulier ?
9. Une formation complémentaire est-elle nécessaire ?
10. Une deuxième phase doit-elle être envisagée ?

---

# Mesurer l’adoption

L’adoption ne se mesure pas uniquement au nombre de comptes créés.

Un compte actif ne prouve pas que l’utilisateur maîtrise le processus.

## Signaux d’adoption

- les utilisateurs se connectent régulièrement ;
- les opérations sont réalisées dans KARMDA ;
- les données sont saisies dans les délais ;
- les anciens registres sont progressivement abandonnés ;
- les utilisateurs posent des questions plus précises ;
- les ambassadeurs répondent aux difficultés simples ;
- la direction consulte les rapports ;
- les erreurs diminuent ;
- l’établissement dépend moins de RELAIS pour les tâches courantes.

## Signaux de faible adoption

- comptes jamais utilisés ;
- opérations réalisées hors plateforme ;
- doubles saisies persistantes ;
- utilisateurs attendant que RELAIS fasse les tâches à leur place ;
- questions identiques répétées ;
- refus silencieux de certains groupes ;
- absence de soutien visible de la direction ;
- données rarement mises à jour ;
- forte dépendance à une seule personne.

---

# Comprendre les causes d’une faible adoption

Une faible adoption ne signifie pas automatiquement que les utilisateurs sont résistants.

Elle peut provenir de :

- formation insuffisante ;
- processus trop complexe ;
- manque d’appareils ;
- connexion instable ;
- rôles mal configurés ;
- données incorrectes ;
- responsabilités imprécises ;
- absence d’instructions de la direction ;
- peur de l’erreur ;
- habitude de l’ancien système ;
- fonctionnalité mal adaptée ;
- surcharge de travail ;
- calendrier de lancement défavorable.

Avant de demander plus d’efforts aux utilisateurs, RELAIS doit identifier l’obstacle réel.

---

# Corriger sans créer de dépendance

Pendant la stabilisation, RELAIS doit aider rapidement.

Mais il ne doit pas réaliser indéfiniment les tâches à la place de l’établissement.

## Mauvaise pratique

À chaque difficulté, RELAIS se connecte et effectue directement l’action pour l’utilisateur.

## Bonne pratique

RELAIS :

1. comprend le problème ;
2. explique la solution ;
3. accompagne l’utilisateur pendant qu’il réalise l’action ;
4. vérifie sa compréhension ;
5. documente la réponse ;
6. laisse l’utilisateur reproduire l’action.

> **Aider un utilisateur à réussir vaut mieux que réussir à sa place.**

---

# Organiser les rappels de formation

Les difficultés observées doivent orienter les rappels.

Un rappel de formation doit être :

- court ;
- ciblé ;
- pratique ;
- lié à une tâche réelle ;
- destiné aux utilisateurs concernés ;
- accompagné d’un support simple.

## Exemples

- rappel de 20 minutes sur la saisie des présences ;
- fiche d’une page sur la réinitialisation du mot de passe ;
- vidéo courte sur la création d’une évaluation ;
- démonstration pratique pour les administrateurs ;
- session spéciale pour les utilisateurs absents.

Il n’est pas nécessaire de répéter toute la formation lorsque seule une étape pose problème.

---

# Renforcer les ambassadeurs internes

Les ambassadeurs doivent recevoir :

- des réponses claires ;
- des supports à partager ;
- un canal d’escalade ;
- la reconnaissance de leur rôle ;
- des limites précises.

Ils ne doivent pas :

- partager leurs identifiants ;
- modifier des données sans autorisation ;
- inventer des procédures ;
- contourner les permissions ;
- promettre des fonctionnalités ;
- devenir l’unique point de dépendance de l’établissement.

Un bon ambassadeur renforce l’autonomie collective.

Il ne remplace pas l’organisation.

---

# Contrôler la qualité des nouvelles données

Après le lancement, les utilisateurs commencent à créer de nouvelles données.

Il faut vérifier que ces informations restent fiables.

## Contrôles possibles

- présences saisies aux bonnes dates ;
- élèves affectés aux bonnes classes ;
- doublons créés après le lancement ;
- notes enregistrées dans les bonnes évaluations ;
- paiements liés aux bons dossiers ;
- utilisateurs créés avec les bons rôles ;
- informations obligatoires complétées ;
- corrections documentées.

Une bonne importation initiale peut être rapidement dégradée par de mauvaises pratiques.

La qualité des données doit devenir une responsabilité permanente de l’établissement.

---

# Surveiller les anciens processus

Après la mise en production, certains utilisateurs peuvent continuer à utiliser :

- registres papier ;
- fichiers Excel ;
- groupes WhatsApp ;
- carnets personnels ;
- ancien logiciel ;
- procédures orales.

Une période parallèle peut être volontaire pendant la transition.

Elle doit toutefois avoir :

- une raison ;
- un responsable ;
- une durée ;
- une méthode de comparaison ;
- une date de fin.

Sans cela, l’établissement risque de maintenir plusieurs versions contradictoires de la vérité.

---

# Gérer les demandes d’évolution

Les utilisateurs découvrent souvent de nouvelles idées après le lancement.

Ces idées sont utiles, mais elles doivent être gérées séparément du support.

Pour chaque demande, documenter :

- le besoin réel ;
- l’utilisateur concerné ;
- la fréquence du problème ;
- la solution actuelle ;
- l’impact attendu ;
- le lien avec le périmètre ;
- le niveau de priorité ;
- la décision produit ou commerciale.

RELAIS ne doit pas promettre immédiatement une nouvelle fonctionnalité.

La bonne réponse peut être :

> Votre demande est bien comprise. Nous allons la documenter et l’évaluer afin de déterminer si elle peut être intégrée au produit, proposée dans une future phase ou traitée par une autre solution.

---

# Communiquer les progrès

Durant la stabilisation, l’établissement doit recevoir une visibilité régulière.

## Exemple de résumé

```text
Période : 22 au 28 juillet

Incidents signalés : 12
Incidents résolus : 10
Incidents ouverts : 2
Problèmes critiques : 0

Utilisateurs actifs : 38 sur 45
Classes utilisant les présences : 14 sur 16

Actions principales :
- correction de trois affectations d’enseignants ;
- rappel de formation sur les absences justifiées ;
- activation de cinq comptes administratifs ;
- vérification des notifications aux parents.

Priorités de la semaine suivante :
- accompagner les deux classes restantes ;
- finaliser les comptes parents ;
- réduire les doubles saisies papier/KARMDA.
```

Un bon suivi montre les progrès sans cacher les difficultés.

---

# Escalader un problème

Un problème doit être escaladé lorsqu’il :

- dépasse les compétences de la personne en charge ;
- affecte plusieurs établissements ;
- concerne la sécurité ;
- menace l’intégrité des données ;
- reste non résolu au-delà du délai attendu ;
- nécessite une décision de direction ;
- peut affecter la réputation de RELAIS ;
- provoque une interruption importante.

L’escalade n’est pas un échec.

Elle permet d’impliquer la bonne personne au bon moment.

---

# Réaliser le bilan de déploiement

À la fin de la période de stabilisation, RELAIS doit organiser un bilan avec l’établissement.

## Le bilan doit couvrir

- les objectifs initiaux ;
- les résultats obtenus ;
- les modules réellement utilisés ;
- les incidents rencontrés ;
- les problèmes restant ouverts ;
- les actions de formation réalisées ;
- le niveau d’adoption ;
- les retours des utilisateurs ;
- les améliorations possibles ;
- les responsabilités futures ;
- le passage au support régulier.

## Questions à poser

- Qu’est-ce qui fonctionne bien ?
- Qu’est-ce qui reste difficile ?
- Quels bénéfices sont déjà visibles ?
- Quels utilisateurs ont encore besoin d’aide ?
- Quels anciens processus subsistent ?
- Quelles responsabilités doivent être renforcées ?
- Quelles améliorations sont prioritaires ?
- L’établissement se sent-il capable de poursuivre avec le support normal ?

---

# Définir la fin du déploiement

Un déploiement ne doit pas rester indéfiniment ouvert.

Il doit disposer de conditions de clôture.

La phase de déploiement peut être clôturée lorsque :

- les objectifs initiaux essentiels sont atteints ;
- les processus inclus sont utilisés ;
- aucun incident critique n’est ouvert ;
- les problèmes restants sont documentés et maîtrisés ;
- les utilisateurs prioritaires sont opérationnels ;
- les ambassadeurs ou référents internes sont actifs ;
- les données principales sont fiables ;
- l’établissement connaît les canaux de support ;
- les responsabilités futures sont comprises ;
- le bilan a été réalisé ;
- le passage vers le support régulier est validé.

Clôturer le déploiement ne signifie pas abandonner le client.

Cela signifie que la relation change de mode :

```text
Déploiement renforcé
        ↓
Stabilisation
        ↓
Support régulier
        ↓
Amélioration continue
```

---

# Passage vers le support régulier

Avant la transition, confirmer :

- le canal de support ;
- les horaires ;
- les délais de réponse ;
- les personnes autorisées à contacter RELAIS ;
- les responsabilités de l’établissement ;
- les services inclus dans l’abonnement ;
- les interventions facturables ;
- le traitement des demandes d’évolution ;
- le processus d’escalade ;
- les prochains points de suivi éventuels.

L’établissement doit comprendre la différence entre :

- support ;
- formation supplémentaire ;
- correction de données ;
- administration déléguée ;
- développement personnalisé ;
- nouvelle phase de déploiement.

---

# Documenter les leçons apprises

Chaque déploiement doit améliorer le suivant.

À la fin du projet, l’équipe RELAIS doit répondre à ces questions :

1. Qu’est-ce qui a bien fonctionné ?
2. Qu’est-ce qui a pris plus de temps que prévu ?
3. Quels risques avaient été correctement anticipés ?
4. Quels problèmes auraient pu être évités ?
5. Quels documents doivent être améliorés ?
6. Quels modèles de données doivent être modifiés ?
7. Quelles questions de formation reviennent souvent ?
8. Quelles promesses commerciales doivent être clarifiées ?
9. Quelle amélioration du produit faciliterait les futurs déploiements ?
10. Que ferions-nous différemment la prochaine fois ?

Les apprentissages doivent être ajoutés à la documentation RELAIS.

Ils ne doivent pas rester uniquement dans la mémoire des personnes.

---

# Erreurs fréquentes

## Considérer le lancement comme la fin

Les premières semaines déterminent souvent la perception durable de KARMDA.

---

## Attendre que le client signale tous les problèmes

Certains utilisateurs abandonnent silencieusement.

RELAIS doit vérifier activement l’utilisation.

---

## Traiter toutes les demandes comme urgentes

Une bonne classification protège les vraies priorités.

---

## Corriger sans expliquer

Une correction rapide qui ne renforce pas l’utilisateur crée de la dépendance.

---

## Confondre incident et demande d’évolution

Cette confusion surcharge le support et crée des attentes irréalistes.

---

## Laisser les solutions temporaires devenir permanentes

Toute mesure temporaire doit avoir une date de fin.

---

## Mesurer uniquement les connexions

L’adoption se mesure par les processus réellement réalisés.

---

## Ignorer les anciens outils

Ils peuvent continuer à produire des données contradictoires.

---

## Fermer le déploiement sans bilan

Sans bilan, RELAIS perd l’occasion d’apprendre et le client ne sait pas clairement que la phase est terminée.

---

## Maintenir un accompagnement renforcé indéfiniment

Le but du suivi est l’autonomie, pas la dépendance.

---

# Bonnes pratiques RELAIS

✔ Planifier le suivi avant la mise en production.

✔ Répondre rapidement, même lorsque la solution n’est pas encore disponible.

✔ Classer les incidents selon leur impact réel.

✔ Demander des signalements précis.

✔ Documenter les solutions temporaires.

✔ Vérifier l’utilisation réelle des processus.

✔ Mesurer l’adoption, pas seulement les connexions.

✔ Accompagner l’utilisateur pendant qu’il agit.

✔ Organiser des rappels de formation ciblés.

✔ Renforcer les ambassadeurs internes.

✔ Contrôler la qualité des nouvelles données.

✔ Suivre la disparition progressive des anciens processus.

✔ Séparer le support des demandes d’évolution.

✔ Communiquer régulièrement les progrès.

✔ Réaliser un bilan formel.

✔ Documenter les leçons apprises.

✔ Préparer clairement le passage vers le support régulier.

# Livrables du suivi post-déploiement

À la fin de cette phase, le dossier de déploiement doit contenir :

- le plan de stabilisation ;
- le registre des incidents ;
- les rapports de suivi ;
- les corrections réalisées ;
- les solutions temporaires utilisées ;
- les supports de rappel de formation ;
- les indicateurs d’adoption ;
- les retours des utilisateurs ;
- le registre des demandes d’évolution ;
- le bilan de déploiement ;
- les problèmes restant ouverts ;
- le plan de passage au support ;
- la validation de clôture ;
- les leçons apprises.

---

# Conditions de clôture

Le déploiement peut être officiellement clôturé lorsque :

- l’établissement utilise les processus prioritaires ;
- les utilisateurs essentiels sont autonomes ;
- aucun incident critique ne reste ouvert ;
- les problèmes résiduels possèdent un responsable et une échéance ;
- les données principales restent fiables ;
- les canaux de support sont connus ;
- les responsabilités sont transférées ;
- le bilan a été partagé ;
- la direction de l’établissement accepte la transition ;
- RELAIS a documenté les apprentissages du projet.

---

# Ce qu’il faut retenir

À l’issue de ce chapitre, retenez les idées suivantes :

- la mise en production commence l’utilisation réelle, elle ne termine pas le déploiement ;
- les premiers jours nécessitent un accompagnement renforcé ;
- les incidents doivent être classés et documentés ;
- le client doit recevoir de la visibilité pendant leur traitement ;
- l’adoption se mesure par les activités réalisées ;
- une faible adoption doit être comprise avant d’être jugée ;
- RELAIS doit aider les utilisateurs sans créer de dépendance ;
- les solutions temporaires doivent rester temporaires ;
- le déploiement doit se terminer par un bilan et une transition claire ;
- chaque projet doit améliorer la méthode RELAIS.

## Question de réflexion

Cependant :

- seulement la moitié des enseignants enregistre régulièrement les présences ;
- plusieurs utilisateurs continuent à utiliser les registres papier ;
- le responsable interne demande souvent à RELAIS de réaliser les corrections à sa place ;
- la direction affirme que le système fonctionne et souhaite clôturer immédiatement le déploiement.
