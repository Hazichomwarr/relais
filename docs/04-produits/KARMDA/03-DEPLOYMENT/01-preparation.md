# KARMDA

# 01 - Préparation du déploiement

## Déploiement RELAIS

> Ce document fait partie du programme officiel de déploiement KARMDA.

**Objectif**

À la fin de cette lecture, vous serez capable de préparer un déploiement KARMDA en définissant clairement ses objectifs, son périmètre, ses responsabilités, son calendrier et ses conditions de réussite.

**Temps estimé**

25 minutes

**Prérequis**

README.md

---

# Pourquoi ce chapitre est important

Un déploiement ne commence pas au moment où un établissement est créé dans KARMDA.

Il commence au moment où RELAIS et l'établissement s'accordent sur ce qui doit être accompli.

Sans préparation, les difficultés apparaissent rapidement :

- les attentes restent imprécises ;
- les responsabilités ne sont pas comprises ;
- les informations arrivent en retard ;
- le calendrier devient irréaliste ;
- certaines demandes sont découvertes trop tard ;
- chacun pense que l'autre devait agir.

La préparation permet de transformer une promesse commerciale en projet concret, organisé et réalisable.

---

# Notre philosophie

Chez RELAIS, nous ne commençons pas un déploiement en posant la question :

> Que devons-nous configurer dans KARMDA ?

Nous commençons par demander :

> Que doit permettre ce déploiement à l'établissement ?

Cette distinction est essentielle.

Le logiciel est un moyen.

La réussite de l'établissement est l'objectif.

---

# Les objectifs de la préparation

La phase de préparation doit permettre de répondre clairement aux questions suivantes :

1. Pourquoi l'établissement adopte-t-il KARMDA ?
2. Quels problèmes souhaite-t-il résoudre en priorité ?
3. Quelles fonctionnalités seront déployées ?
4. Quelles fonctionnalités ne font pas partie du déploiement actuel ?
5. Qui participe au projet ?
6. Qui prend les décisions ?
7. Qui fournit les données ?
8. Quel est le calendrier prévu ?
9. Quels risques pourraient retarder le projet ?
10. Comment déterminerons-nous que le déploiement est réussi ?

Aucune installation ne doit commencer tant que les réponses essentielles restent inconnues.

---

# Confirmer la transition entre la vente et le déploiement

Avant de contacter l'établissement pour lancer le projet, l'équipe de déploiement doit recevoir les informations transmises par l'équipe commerciale.

Cette transition doit inclure au minimum :

- le nom de l'établissement ;
- les coordonnées du contact principal ;
- l'offre souscrite ;
- les fonctionnalités incluses ;
- les services complémentaires convenus ;
- les engagements pris durant la vente ;
- le calendrier évoqué avec le client ;
- les besoins particuliers identifiés ;
- les éventuelles réserves ou préoccupations du client ;
- les documents commerciaux signés.

L'équipe de déploiement ne doit jamais découvrir les engagements commerciaux devant le client.

Toute promesse ambiguë doit être clarifiée en interne avant la réunion de lancement.

---

# Définir les raisons du projet

Chaque établissement adopte KARMDA pour des raisons différentes.

Par exemple :

- mieux suivre les présences ;
- produire les bulletins plus rapidement ;
- centraliser les dossiers des élèves ;
- améliorer la communication avec les parents ;
- suivre les paiements ;
- disposer de statistiques fiables ;
- réduire l'utilisation du papier ;
- structurer une administration en croissance.

Ces motivations doivent être documentées.

Elles serviront ensuite à :

- définir les priorités ;
- orienter la configuration ;
- adapter les formations ;
- mesurer la réussite ;
- éviter de présenter des fonctionnalités sans rapport avec les besoins réels.

---

# Définir les objectifs du déploiement

Un objectif doit décrire un résultat concret attendu par l'établissement.

## Exemple trop vague

> Améliorer la gestion de l'école.

Cet objectif ne permet pas de mesurer la réussite.

## Exemple plus précis

> Permettre à l'administration d'inscrire tous les élèves dans KARMDA et aux enseignants d'enregistrer les présences quotidiennes à partir du début du prochain trimestre.

Cet objectif précise :

- les utilisateurs concernés ;
- le processus visé ;
- le résultat attendu ;
- la période de mise en œuvre.

---

# Définir le périmètre

Le périmètre indique ce qui fait partie du déploiement.

Il protège à la fois l'établissement et RELAIS contre les malentendus.

## Éléments pouvant faire partie du périmètre

- configuration de l'établissement ;
- création des années scolaires ;
- création des classes et sections ;
- import des élèves ;
- création des comptes utilisateurs ;
- gestion des présences ;
- gestion des notes ;
- génération des bulletins ;
- suivi financier ;
- communication avec les parents ;
- formation des utilisateurs ;
- accompagnement après lancement.

## Éléments hors périmètre

Les éléments non inclus doivent également être identifiés.

Par exemple :

- développement d'une fonctionnalité personnalisée ;
- nettoyage complet des archives papier ;
- saisie manuelle de plusieurs années de données historiques ;
- fourniture d'ordinateurs ou de connexion Internet ;
- intégration avec un système externe non prévue au contrat ;
- formation supplémentaire non incluse dans l'offre.

Dire clairement ce qui n'est pas inclus évite les déceptions futures.

---

# Identifier les parties prenantes

Un déploiement réussi nécessite des interlocuteurs clairement identifiés.

## Le sponsor du projet

Le sponsor est généralement un membre de la direction.

Il soutient le projet, facilite les décisions et intervient lorsque des obstacles importants apparaissent.

---

## Le responsable de l'établissement

Il coordonne le projet du côté de l'école.

Il doit pouvoir :

- fournir les informations nécessaires ;
- mobiliser les utilisateurs ;
- valider les configurations ;
- suivre les échéances ;
- transmettre les décisions de l'établissement.

---

## Le chef de projet RELAIS

Il coordonne le déploiement du côté de RELAIS.

Il est responsable de :

- l'organisation des étapes ;
- la communication avec l'établissement ;
- le suivi du calendrier ;
- l'identification des risques ;
- la coordination des équipes internes ;
- la documentation des décisions.

---

## Les référents fonctionnels

Ce sont les personnes qui connaissent les opérations quotidiennes de l'établissement.

Il peut s'agir :

- du responsable des inscriptions ;
- du responsable pédagogique ;
- du comptable ;
- du surveillant général ;
- d'un enseignant référent ;
- du responsable informatique.

Ils aident RELAIS à comprendre les pratiques réelles de l'établissement.

---

## Les utilisateurs finaux

Les utilisateurs finaux sont les personnes qui utiliseront KARMDA au quotidien :

- direction ;
- administration ;
- enseignants ;
- parents ;
- élèves, lorsque leur accès est prévu.

Leurs réalités doivent être prises en compte dès la préparation.

---

# Clarifier les responsabilités

Chaque tâche importante doit avoir un responsable.

| Activité                                  | RELAIS                                 | Établissement       |
| ----------------------------------------- | -------------------------------------- | ------------------- |
| Organiser le calendrier du déploiement    | Responsable                            | Participe et valide |
| Fournir les informations officielles      | Vérifie                                | Responsable         |
| Préparer les fichiers d'import            | Accompagne et contrôle                 | Fournit les données |
| Configurer KARMDA                         | Responsable                            | Valide              |
| Identifier les utilisateurs               | Accompagne                             | Responsable         |
| Créer les accès                           | Responsable selon le processus convenu | Valide              |
| Mobiliser les participants aux formations | Informe                                | Responsable         |
| Former les utilisateurs                   | Responsable                            | Participe           |
| Valider la mise en production             | Recommande                             | Décide avec RELAIS  |
| Signaler les difficultés                  | Analyse et accompagne                  | Informe rapidement  |

Une responsabilité partagée sans responsable clairement désigné devient souvent une responsabilité oubliée.

---

# Établir le calendrier

Le calendrier doit être réaliste et validé par les deux parties.

Il peut contenir les étapes suivantes :

| Étape                     | Date prévue | Responsable             | Statut       |
| ------------------------- | ----------- | ----------------------- | ------------ |
| Réunion de lancement      | À définir   | RELAIS et établissement | Non commencé |
| Collecte des informations | À définir   | Établissement           | Non commencé |
| Configuration initiale    | À définir   | RELAIS                  | Non commencé |
| Préparation des données   | À définir   | Établissement           | Non commencé |
| Import et vérification    | À définir   | RELAIS                  | Non commencé |
| Formation                 | À définir   | RELAIS et établissement | Non commencé |
| Validation finale         | À définir   | Les deux parties        | Non commencé |
| Mise en production        | À définir   | Les deux parties        | Non commencé |
| Suivi post-lancement      | À définir   | RELAIS                  | Non commencé |

Une date de lancement ne doit pas être annoncée sans tenir compte :

- de la disponibilité des équipes ;
- de la qualité des données ;
- des périodes d'examen ;
- des vacances scolaires ;
- des inscriptions ;
- de la rentrée scolaire ;
- de la disponibilité des responsables ;
- du temps nécessaire à la formation.

---

# Déterminer les informations à collecter

Avant l'installation, RELAIS doit établir la liste des informations nécessaires.

## Informations institutionnelles

- nom officiel ;
- logo ;
- adresse ;
- téléphone ;
- adresse e-mail ;
- responsables ;
- coordonnées administratives.

## Informations académiques

- année scolaire ;
- cycles ;
- niveaux ;
- classes ;
- sections ;
- matières ;
- périodes académiques ;
- système de notation ;
- règles de passage ;
- calendrier scolaire.

## Informations sur les utilisateurs

- direction ;
- administrateurs ;
- personnel administratif ;
- enseignants ;
- parents ;
- élèves.

## Informations financières

Lorsque le module concerné est inclus :

- frais de scolarité ;
- catégories de frais ;
- échéances ;
- modes de paiement ;
- remises éventuelles ;
- règles de suivi.

## Informations techniques

- disponibilité d'Internet ;
- équipements utilisés ;
- adresses e-mail disponibles ;
- personnes pouvant fournir une assistance locale ;
- éventuelles contraintes d'accès.

La liste exacte dépend toujours du périmètre convenu.

---

# Vérifier les prérequis de l'établissement

Avant de poursuivre, l'établissement doit disposer des conditions minimales nécessaires à l'utilisation de KARMDA.

Il faut notamment vérifier :

- l'accès à Internet ;
- la disponibilité des appareils nécessaires ;
- la capacité des utilisateurs à accéder à leurs comptes ;
- l'existence d'un responsable interne ;
- la disponibilité des données ;
- la disponibilité des participants aux formations ;
- l'engagement de la direction.

Une difficulté technique peut être corrigée.

Un manque d'engagement non identifié peut compromettre tout le projet.

---

# Identifier les risques

Un risque est un événement susceptible de retarder ou de fragiliser le déploiement.

## Risques fréquents

- données incomplètes ;
- fichiers mal organisés ;
- interlocuteur indisponible ;
- décisions tardives ;
- résistance des utilisateurs ;
- connexion Internet instable ;
- calendrier trop ambitieux ;
- absence de soutien de la direction ;
- changements de périmètre ;
- période scolaire défavorable.

Pour chaque risque important, il faut documenter :

- sa nature ;
- sa probabilité ;
- son impact ;
- la personne chargée de le suivre ;
- l'action prévue pour le réduire.

## Exemple

| Risque                                      | Impact                             | Mesure préventive                                                      |
| ------------------------------------------- | ---------------------------------- | ---------------------------------------------------------------------- |
| Liste des élèves reçue en retard            | Retard de l'import et du lancement | Fixer une date limite et fournir le modèle de fichier dès le démarrage |
| Enseignants indisponibles pour la formation | Faible adoption                    | Planifier les sessions avec la direction avant la configuration finale |
| Connexion Internet instable                 | Difficultés d'utilisation          | Tester l'accès et prévoir les conditions d'utilisation adaptées        |
| Périmètre mal compris                       | Insatisfaction du client           | Faire valider par écrit les éléments inclus et exclus                  |

---

# Définir le plan de communication

Le client ne doit jamais se demander ce qui se passe.

Avant le début du projet, il faut préciser :

- le contact principal chez RELAIS ;
- le contact principal dans l'établissement ;
- le canal de communication officiel ;
- la fréquence des points de suivi ;
- la manière de signaler un blocage ;
- la personne autorisée à valider les décisions ;
- les informations qui doivent être confirmées par écrit.

Les échanges par téléphone ou messagerie peuvent faciliter le travail.

Cependant, toute décision importante doit être documentée.

---

# Définir les critères de réussite

La réussite doit être mesurable.

Les critères peuvent inclure :

- tous les élèves actifs sont présents dans KARMDA ;
- les classes et sections ont été validées ;
- les utilisateurs prioritaires disposent de leurs accès ;
- les enseignants formés peuvent saisir une présence ;
- l'administration peut inscrire un nouvel élève ;
- la direction peut consulter les indicateurs prévus ;
- les données importées ont été contrôlées ;
- aucun problème bloquant ne subsiste ;
- l'établissement a validé sa mise en production.

Les critères doivent correspondre aux objectifs et au périmètre du projet.

---

# Constituer le dossier de déploiement

Chaque établissement doit disposer d'un dossier de déploiement organisé.

Ce dossier peut contenir :

- le contrat ou bon de commande ;
- la fiche de transition commerciale ;
- le périmètre validé ;
- les coordonnées des parties prenantes ;
- le calendrier ;
- la liste des informations demandées ;
- les fichiers reçus ;
- les décisions prises ;
- les risques identifiés ;
- les validations du client ;
- les comptes rendus de réunion ;
- la checklist de mise en production ;
- le bilan post-lancement.

Le dossier constitue la mémoire du projet.

Il permet à une autre personne de comprendre le déploiement sans dépendre uniquement des souvenirs de l'équipe.

---

# Réunion interne de préparation

Avant la réunion de lancement avec l'établissement, l'équipe RELAIS doit réaliser un court point interne.

Cette réunion permet de vérifier :

- ce qui a été vendu ;
- ce qui doit être livré ;
- les responsabilités internes ;
- les questions à poser au client ;
- les risques déjà identifiés ;
- les éventuelles ambiguïtés contractuelles ;
- la capacité de respecter le calendrier annoncé.

RELAIS doit se présenter devant le client avec une compréhension commune du projet.

---

# Conditions nécessaires avant l'installation

L'installation peut commencer lorsque les conditions suivantes sont réunies :

- le contrat ou l'accord commercial est confirmé ;
- le périmètre est compris ;
- le responsable du projet est identifié de chaque côté ;
- les objectifs prioritaires sont définis ;
- les informations institutionnelles essentielles sont disponibles ;
- le calendrier initial est établi ;
- les principaux risques sont connus ;
- les responsabilités sont clarifiées ;
- la prochaine étape est acceptée par les deux parties.

Lorsque certaines informations secondaires manquent, le projet peut parfois avancer.

En revanche, il ne doit jamais commencer sur des objectifs, un périmètre ou des responsabilités ambigus.

---

# Erreurs fréquentes

## Commencer parce que le client est pressé

L'urgence du client ne doit pas devenir du désordre pour le projet.

---

## Confondre date souhaitée et date réaliste

Une date demandée doit être analysée avant d'être confirmée.

---

## Ne pas documenter les promesses commerciales

Une promesse oubliée par RELAIS reste une promesse attendue par le client.

---

## Accepter un périmètre imprécis

Les expressions comme « tout configurer » ou « mettre toute l'école dans le système » doivent être précisées.

---

## Dépendre d'une seule personne

Le projet devient fragile lorsqu'une seule personne possède toutes les informations ou toutes les décisions.

---

## Reporter la question des données

La qualité et la disponibilité des données doivent être évaluées dès le début.

---

## Penser uniquement au logiciel

Un déploiement dépend autant des personnes, des décisions et de l'organisation que de la technologie.

---

# Bonnes pratiques RELAIS

✔ Lire les documents commerciaux avant de rencontrer le client.

✔ Reformuler les objectifs avec des mots simples.

✔ Faire valider le périmètre par écrit.

✔ Attribuer chaque responsabilité à une personne identifiable.

✔ Documenter les décisions importantes.

✔ Annoncer rapidement les risques.

✔ Ne jamais confirmer une date irréaliste pour satisfaire momentanément le client.

✔ Adapter le déploiement à la réalité de l'établissement.

✔ Préparer la réussite de l'utilisateur, pas seulement le fonctionnement du logiciel.

---

# Livrables de la phase de préparation

À la fin de cette phase, le dossier de déploiement doit contenir au minimum :

- une fiche de présentation de l'établissement ;
- les objectifs du projet ;
- le périmètre inclus et exclu ;
- la liste des parties prenantes ;
- la répartition des responsabilités ;
- le calendrier prévisionnel ;
- la liste des données et documents attendus ;
- les principaux risques ;
- les critères de réussite ;
- le plan de communication ;
- la confirmation de passage à l'installation.

---

# Ce qu'il faut retenir

À l'issue de ce chapitre, retenez les idées suivantes :

- le déploiement commence avant toute configuration technique ;
- les objectifs de l'établissement guident le projet ;
- le périmètre doit être clair ;
- chaque responsabilité doit être attribuée ;
- le calendrier doit refléter la réalité ;
- les risques doivent être identifiés tôt ;
- les décisions importantes doivent être documentées ;
- l'installation commence uniquement lorsque le projet est suffisamment compris.

---

## Phrase à retenir

> **« Un projet bien préparé ne supprime pas tous les obstacles, mais il évite qu'ils deviennent des surprises. »**

## Question de réflexion

Le commercial a promis à un établissement une mise en production dans deux semaines.

Lors de la préparation, vous découvrez que les données des élèves ne sont pas prêtes, que les enseignants ne seront disponibles pour la formation que dans trois semaines et que le périmètre du projet reste imprécis.

Comment gérez-vous cette situation sans désavouer le commercial, sans accuser le client et sans engager RELAIS sur un calendrier irréaliste ?
