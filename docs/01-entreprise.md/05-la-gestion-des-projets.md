# RELAIS

# La gestion des projets

> **On prend le relais.**
>
> _Pour que vous puissiez avancer plus facilement._

---

# Pourquoi ce document existe

Chaque projet représente une promesse.

Une promesse faite à un client.

Une promesse faite à notre équipe.

Une promesse faite à nous-mêmes.

Chez RELAIS, nous refusons les projets improvisés.

Nous préférons consacrer davantage de temps à préparer un projet qu'à corriger les conséquences d'une mauvaise préparation.

Notre objectif n'est pas seulement de livrer.

Notre objectif est de livrer un projet dont nous pouvons être fiers.

---

# Notre philosophie

Nous ne gérons pas des tâches.

Nous accompagnons un projet depuis une idée jusqu'à une solution qui crée de la valeur.

Un projet ne se mesure pas au nombre de fonctionnalités développées.

Il se mesure à l'impact qu'il produit chez notre client.

---

# Le cycle de vie d'un projet

Chaque projet suit toujours les mêmes étapes.

```
Découverte

↓

Qualification

↓

Documentation

↓

Conception

↓

Validation

↓

Développement

↓

Tests

↓

Livraison

↓

Accompagnement

↓

Amélioration continue
```

Aucune étape n'est ignorée.

Chaque étape prépare la suivante.

---

# 1. Découverte

Tout projet commence par une rencontre.

Nous cherchons avant tout à comprendre.

Nous voulons connaître :

- le métier du client ;
- ses objectifs ;
- ses difficultés ;
- ses habitudes de travail ;
- les outils déjà utilisés.

À ce stade, nous ne proposons encore aucune solution.

Nous écoutons.

---

# 2. Qualification

Tous les projets ne doivent pas être acceptés.

Avant de nous engager, nous vérifions :

- que le besoin est clairement identifié ;
- que nous sommes capables d'y répondre ;
- que le calendrier est réaliste ;
- que le budget est cohérent ;
- que la collaboration est saine.

Savoir dire non fait partie de notre professionnalisme.

---

# 3. Documentation

Avant d'écrire une ligne de code, nous écrivons des documents.

Selon la nature du projet, cela peut inclure :

- cahier des besoins ;
- modèle métier ;
- architecture ;
- maquettes ;
- parcours utilisateur ;
- planning ;
- estimation.

Une documentation de qualité réduit considérablement les risques.

---

# 4. Conception

Nous concevons avant de produire.

Chaque décision technique doit répondre à une décision métier.

Nous recherchons :

- la simplicité ;
- la cohérence ;
- l'évolutivité ;
- la maintenabilité.

Le meilleur design est celui qui reste compréhensible plusieurs années plus tard.

---

# 5. Validation

Avant le développement, nous validons avec le client.

Nous nous assurons que chacun partage la même compréhension du projet.

Cette étape évite les mauvaises surprises.

Changer une idée coûte peu.

Changer un logiciel terminé coûte cher.

---

# 6. Développement

Le développement est une phase d'exécution.

Le code traduit une conception déjà validée.

Pendant cette étape :

- nous développons progressivement ;
- nous documentons lorsque nécessaire ;
- nous testons régulièrement ;
- nous privilégions la qualité à la vitesse.

---

# 7. Tests

Avant toute livraison, nous vérifions :

- les fonctionnalités ;
- les cas particuliers ;
- les performances ;
- la sécurité ;
- l'expérience utilisateur.

Un projet n'est prêt que lorsqu'il inspire confiance.

---

# 8. Livraison

La livraison n'est jamais la fin du projet.

Elle marque le début de son utilisation réelle.

Chaque livraison comprend :

- la mise en production ;
- les vérifications finales ;
- la documentation nécessaire ;
- la formation du client lorsque cela est prévu.

Nous ne livrons jamais un produit que nous ne serions pas prêts à utiliser nous-mêmes.

---

# 9. Accompagnement

Un client n'achète pas seulement un logiciel.

Il choisit un partenaire.

Après la livraison, nous restons disponibles pour :

- répondre aux questions ;
- accompagner la prise en main ;
- corriger les éventuels incidents ;
- recueillir les retours.

La réussite d'un projet se mesure aussi après sa mise en production.

---

# 10. Amélioration continue

Chaque projet nous apprend quelque chose.

À la fin de chaque mission, nous analysons :

- ce qui a bien fonctionné ;
- ce qui aurait pu être amélioré ;
- ce que nous devons documenter ;
- ce que nous devons standardiser.

Chaque projet rend RELAIS plus expérimentée que le précédent.

---

# Les demandes de changement

Les besoins évoluent.

C'est normal.

Toute demande de modification est analysée selon trois critères :

- sa valeur métier ;
- son impact technique ;
- son impact sur les délais et le budget.

Nous expliquons toujours les conséquences d'une modification avant de la mettre en œuvre.

La transparence protège la relation avec nos clients.

---

# Notre définition d'un projet réussi

Un projet est réussi lorsque :

- le client atteint ses objectifs ;
- les utilisateurs adoptent naturellement la solution ;
- le logiciel est fiable ;
- l'équipe est fière du résultat ;
- le projet peut évoluer sereinement.

La livraison seule n'est jamais un indicateur suffisant.

---

# Notre engagement

Chaque projet est traité avec la même rigueur, quelle que soit sa taille.

Qu'il s'agisse d'un site web, d'un logiciel métier ou d'une plateforme nationale,

nous appliquons les mêmes principes.

Comprendre.

Concevoir.

Construire.

Accompagner.

Améliorer.

Car chez RELAIS, un projet n'est pas une succession de tâches.

C'est un engagement envers la réussite de notre client.
